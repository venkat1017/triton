Why the memory space matters more than the operation

Here is something most engineers don't realize until they've wasted weeks chasing the wrong optimizations: for the vast majority of ML kernels, the bottleneck is not the computation — it's the memory access. A GPU doing a billion multiplications per second is useless if it spends 99% of its time waiting for data to arrive from DRAM.

This is not a defect. It's a consequence of arithmetic intensity — the ratio of compute to memory traffic — being low for most operations you care about. Layer normalization, softmax, elementwise activations: these move bytes and do almost no math per byte. The hardware can crunch numbers far faster than it can fetch them. So the number that matters is not how much math you're doing, it's where that data lives when you do it.

The GPU exposes four memory spaces with dramatically different properties. Choosing the right one — and accessing it correctly — is the entire job of kernel optimization. Not clever algorithms. Not unrolling loops. The right memory space, accessed with the right access pattern.

Lesson 01 showed you the hardware picture: the latency pyramid from registers (1 cycle) through HBM (700 cycles). This lesson is the programmer's view of the same hierarchy: what the four spaces are, how you access them, what goes wrong, and what every fast kernel does to work around the constraints.

Global memory: where all tensors live

Global memory is HBM — the high-bandwidth DRAM physically packaged alongside the GPU die. Every PyTorch tensor you create with .cuda() lives here. The A100 has 80 GB of it. The T4 has 16 GB. When you do x = torch.randn(1024, 1024, device='cuda'), those floats go into global memory. Every kernel you ever write starts by loading from global memory and ends by storing results back to it.

The properties you need to internalize: 700-cycle latency, 2 TB/s peak bandwidth on A100, visible to all SMs, persistent for the lifetime of the tensor. That 700-cycle latency is the enemy. Every optimization in this lesson is a strategy for paying it as rarely as possible.

The bandwidth figure deserves more attention. 2 TB/s sounds enormous. It is enormous — it's the result of putting HBM stacks directly on the GPU package with thousands of parallel data lanes. But compare it to compute: the A100 does 312 TFLOP/s in BF16. At 2 bytes per BF16 element, that's 156 TFLOP per TB, or 156 FLOP per byte. Your kernel must do at least 156 floating-point operations per byte of data it loads, or it can't keep the compute units fed. Most operations don't come close to this threshold. relu(x) does 1 FLOP per 2 bytes: arithmetic intensity of 0.5, which is 312× below the threshold for the A100. Every cycle of compute is preceded by hundreds of cycles of waiting.

The practical consequence of this math: for memory-bound kernels, you don't optimize by doing less compute. You optimize by doing fewer, better memory accesses. The 312 TFLOP/s figure is irrelevant if your kernel is memory-bound. Making the memory access pattern efficient is the entire optimization.

Coalesced memory access: the single most important access pattern

The GPU memory controller has one critical optimization: when all 32 threads in a warp request data from contiguous memory addresses, the controller merges those 32 requests into a single transaction. One memory request. One latency penalty. Full bandwidth utilization.

When the 32 threads instead request data from 32 scattered addresses — different cache lines, different DRAM rows — the controller cannot merge them. It issues up to 32 separate transactions. Same amount of data, 32× the memory traffic, 32× the bandwidth consumed, and in the worst case, 32× the latency.

This is coalescing, and it is the single most impactful access pattern rule in GPU programming. It's not one technique among many. It is the technique, the one that underlies everything else.

The coalescing rule, stated precisely

Thread k in a warp should access memory address base + k × element_size. That's it. If every thread accesses the next consecutive element, the hardware coalesces. Any deviation from this — any stride greater than 1 — begins degrading bandwidth utilization.

The consequence is concrete and shows up in code you write every day. tensor.sum(dim=1) sums along the last dimension. For a row-major tensor, that's reading consecutive elements in the same row — stride 1, coalesced, fast. tensor.sum(dim=0) sums along the first dimension. That's reading the same column across rows — stride N, uncoalesced, slow. On a 4096×4096 float16 tensor on T4, the difference is typically 2.5–3×. Same math, different access pattern, different GPU.

This is why the fix for column-major access is sometimes transposing first, then summing — even though transposing is itself a memory operation. After transposing, the "column" you want is now laid out contiguously in memory. One coalesced scan dominates the cost of the transpose.

The pattern appears in unexpected places

Coalescing matters in any operation where threads access a 2D structure. Attention: when multiple heads read from the same KV cache row, are they reading contiguous positions or jumping? Embedding lookups: if a batch of sequences looks up different token IDs, are consecutive thread IDs looking up consecutive positions in the embedding table? The moment you have a 2D or higher-dimensional tensor, ask: which dimension is varying fastest in memory, and which dimension are my threads varying over?

Shared memory: the programmer-controlled scratchpad

Shared memory is physically inside the SM — on the same piece of silicon as the compute units. On the A100, there is 192 KB of shared memory per SM, configurable as either L1 cache or explicit scratchpad (or a split between the two). The latency is approximately 20 cycles. Not 700 cycles. Not 200 cycles. 20 cycles, which is 35× faster than VRAM for the same data.

The key word is programmer-controlled. Shared memory is not a cache that the hardware fills automatically. You explicitly decide what goes in it, when it goes in, and what computation uses it. This is what makes it powerful and what makes it a discipline: the hardware won't save you if you access it badly.

Shared memory is shared within a thread block. All threads in the same block see the same shared memory. Threads in different blocks on the same SM do not. This is the reason thread block size matters: it determines how many threads can collaborate through shared memory on a piece of work.

The canonical pattern: load once, reuse many times

The canonical use of shared memory is tiling. The idea: instead of letting each thread independently fetch its data from VRAM (each paying 700 cycles per access), have the entire block cooperatively load a tile of data into shared memory once, then let every thread compute from shared memory (20 cycles). The VRAM cost is paid once and amortized over every computation in the tile.

In Triton, you don't explicitly allocate shared memory in most cases. When you call tl.load(), the data lands in registers. When you call tl.dot(), Triton automatically stages the operands through shared memory internally. Understanding why it does this — the 35× latency difference — lets you reason about when tiling helps and when it doesn't, rather than cargo-culting the pattern.

Bank conflicts: how shared memory goes wrong

Shared memory is fast but not uniformly fast. It is divided into 32 banks, each 4 bytes wide, numbered 0 through 31. The bank that holds a given address is determined by: bank = (address_in_bytes / 4) % 32. Each bank services one 4-byte read request per clock cycle. If all 32 threads in a warp access different banks, all 32 reads complete in one cycle. If two threads access different addresses within the same bank, those two requests serialize: 2 cycles instead of 1. An N-way bank conflict multiplies the shared memory latency by N.

The worst case: all 32 threads access different addresses that all map to the same bank. 32 serialized accesses. Shared memory now takes 32 × 20 = 640 cycles for one warp load — worse than L2 cache, nearly as bad as VRAM. All the advantage of placing data in shared memory is gone.

The padding fix, explained precisely

Why does adding one extra column to the shared memory array eliminate bank conflicts? The bank of element [row, col] in a tile of width W is (row × W + col) % 32. When W = 32: element [0, 0] is bank 0, element [1, 0] is bank (1×32 + 0) % 32 = 0, element [2, 0] is bank 0, and so on. Every row's column 0 maps to bank 0. Conflict.

When W = 33 (padded by 1): element [0, 0] is bank 0, element [1, 0] is bank (1×33 + 0) % 32 = 1, element [2, 0] is bank (2×33) % 32 = 2, element [3, 0] is bank (3×33) % 32 = 3. Different bank for each row. Zero conflicts. The padding shifts each row's start address by 4 bytes (one extra element), cycling through bank assignments. This works whenever the tile width is a multiple of 32; the padding breaks the alignment.

The performance cost of padding: you waste 1 float per row of shared memory. For a 128×32 tile, that's 128 × 4 = 512 bytes wasted out of 192 KB. Negligible. The performance gain from eliminating a 32-way bank conflict is substantial. Always pad.

Registers and spilling

Registers are the invisible performance tier. 1-cycle latency, private to each thread, managed entirely by the compiler. You never explicitly allocate registers in Triton or CUDA — the compiler does it based on how many live variables your kernel has at any point.

The A100 has 65,536 32-bit registers per SM. With a maximum of 2,048 threads per SM, that gives 32 registers per thread at maximum occupancy. The hardware supports up to 255 registers per thread, but using more means fewer threads can be resident simultaneously — lower occupancy.

The dangerous scenario is the silent one: register spilling. When the compiler determines your kernel needs more registers than available, it maps the excess to local memory — which is VRAM. The kernel compiles. It runs. There are no errors. It's just slow, and you have no idea why unless you measure the right metric.

In Triton, you can check register usage after compilation:

@triton.jit
def my_kernel(x_ptr, out_ptr, N, BLOCK: tl.constexpr):
    pid  = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    x    = tl.load(x_ptr + offs, mask=offs < N)
    # ... kernel body ...
    tl.store(out_ptr + offs, x, mask=offs < N)

# Trigger compilation by running once
my_kernel[(1,)](x_ptr, out_ptr, N, BLOCK=256)

# Inspect register usage
print(my_kernel.n_regs)    # registers per thread
print(my_kernel.n_spills)  # register spills to local memory

# Rule of thumb for T4 (32 warps × 32 threads = 1024 threads per SM):
# n_regs × 1024 should be ≤ 32768 for full occupancy
# n_spills > 0 means you're paying 700-cycle VRAM costs for register overflows

If n_spills > 0, your options are: reduce the tile size (fewer elements live simultaneously), restructure the kernel to reduce the number of live variables at any point, or accept the lower performance. There is no compiler flag that makes register spills free. The hardware has a fixed number of registers per SM, and you're over budget.

Detect spills with Nsight Compute

To confirm register spills at the hardware level: ncu --metrics l1tex__t_sectors_pipe_lsu_mem_local_op_ld.sum, l1tex__t_sectors_pipe_lsu_mem_local_op_st.sum python my_kernel.py. Any nonzero value for the local load/store sector count means the kernel is spilling to VRAM. For a kernel that should only touch shared memory and registers, a nonzero local sector count is an immediate bug signal.

Local memory: the silent VRAM alias

"Local memory" is the most misleading name in GPU programming. It sounds fast. It is not fast. Local memory is a per-thread region of global VRAM, automatically addressed by the hardware to be private to each thread. It has the same 700-cycle latency as any other global memory access. The word "local" refers to the addressing scope — it's addressed relative to the current thread, not shared with other threads — not to its physical location, which is in DRAM.

Local memory exists for two reasons. The first is register spilling, described above: when the compiler runs out of registers, it maps the overflow to local memory automatically. The second is large thread-local arrays with runtime-variable size: arrays whose size is not a compile-time constant cannot live in registers (register allocation is static), so the compiler places them in local memory. In CUDA C this is a common trap; in Triton it's less common because you operate on fixed-size block tiles.

The important thing to know about local memory is that you cannot optimize it directly. You can't choose an access pattern for it. You can't prefetch it. You can only avoid using it, by reducing register pressure and using only fixed-size tile operations. When you see it appearing in a profile, the diagnosis is always register spills or oversized per-thread arrays, and the fix is always structural: redesign the kernel to need fewer simultaneously live variables.

The canonical tiling pattern

Now that all four memory spaces are defined, here is the pattern that every fast kernel uses. It shows up identically in matrix multiplication, softmax, layer norm, attention, and any other operation where data is reused more than once. The specific operation changes; the structure never does.

Step 1: Partition the input into tiles. Each tile is small enough to fit in shared memory (or registers, for small enough tiles).

Step 2: One thread block is responsible for computing one output tile. The block cooperatively loads the input tiles needed for its output tile from global memory into shared memory. This is one coalesced load from VRAM per tile — paid once.

Step 3: All computation for this output tile reads from shared memory. Every reuse of an input element costs 20 cycles instead of 700. If an element is reused K times, the effective cost drops from 700K to 700 + 20K cycles.

Step 4: The output tile is written from registers back to global memory once. One coalesced store.

The numbers determine when tiling helps. If K = 1 (each input element used exactly once), tiling saves nothing — you're still paying 700 cycles per load. Tiling helps whenever K > 1. For matrix multiplication with BLOCK_K = 64, K = 64 and tiling is 22× cheaper. For softmax with two passes over the same row data, K = 2 and tiling is ~1.4× cheaper. The more reuse, the bigger the payoff.

Two Triton kernels: coalesced vs strided

The most instructive experiment in GPU programming is writing two kernels that are identical in compute but different only in their memory access pattern, then measuring the throughput difference. That's what these two kernels do.

import triton
import triton.language as tl
import torch
from torch.utils import benchmark

# ─── Kernel 1: coalesced row sum ────────────────────────────────────────────
# Each program handles one row of a 2D matrix.
# Threads read contiguous columns within that row → coalesced.
@triton.jit
def row_sum(x_ptr, out_ptr, M, N, BLOCK_N: tl.constexpr):
    row  = tl.program_id(0)               # one program per row
    offs = tl.arange(0, BLOCK_N)           # [0, 1, 2, ..., BLOCK_N-1]
    mask = offs < N
    # Row i starts at byte offset row*N in memory. Consecutive offs → coalesced.
    x    = tl.load(x_ptr + row * N + offs, mask=mask, other=0.0)
    tl.store(out_ptr + row, tl.sum(x, axis=0))


# ─── Kernel 2: uncoalesced column sum ───────────────────────────────────────
# Each program handles one column.
# Threads read the same column across different rows.
# In row-major layout, row k of column j is at offset k*N+j → stride N apart.
@triton.jit
def col_sum(x_ptr, out_ptr, M, N, BLOCK_M: tl.constexpr):
    col  = tl.program_id(0)               # one program per column
    offs = tl.arange(0, BLOCK_M)           # [0, 1, 2, ..., BLOCK_M-1]
    mask = offs < M
    # Element [off, col] is at offset off*N + col. Each step jumps N elements → strided.
    x    = tl.load(x_ptr + offs * N + col, mask=mask, other=0.0)
    tl.store(out_ptr + col, tl.sum(x, axis=0))


# ─── Benchmark ───────────────────────────────────────────────────────────────
M, N  = 4096, 4096
x     = torch.randn(M, N, device='cuda', dtype=torch.float16)
out   = torch.zeros(max(M, N), device='cuda', dtype=torch.float32)

BLOCK = 4096  # must be power of 2 and ≥ N (or M)

t_row = benchmark.Timer(
    stmt="row_sum[(M,)](x, out, M, N, BLOCK)",
    globals=dict(row_sum=row_sum, M=M, N=N, x=x, out=out, BLOCK=BLOCK)
).timeit(500)

t_col = benchmark.Timer(
    stmt="col_sum[(N,)](x, out, M, N, BLOCK)",
    globals=dict(col_sum=col_sum, M=M, N=N, x=x, out=out, BLOCK=BLOCK)
).timeit(500)

bytes_moved = M * N * 2   # read all of x once, float16 = 2 bytes
bw_row = bytes_moved / t_row.mean / 1e9
bw_col = bytes_moved / t_col.mean / 1e9

print(f"Row sum (coalesced):   {bw_row:.0f} GB/s  ({t_row.mean*1e3:.3f} ms)")
print(f"Col sum (uncoalesced): {bw_col:.0f} GB/s  ({t_col.mean*1e3:.3f} ms)")
print(f"Speedup from coalescing: {bw_row/bw_col:.1f}×")

# Verify correctness against PyTorch reference
ref_row = x.float().sum(dim=1)
row_sum[(M,)](x, out, M, N, BLOCK)
print(f"Row correct: {torch.allclose(out[:M], ref_row, atol=1e-2)}")

ref_col = x.float().sum(dim=0)
col_sum[(N,)](x, out, M, N, BLOCK)
print(f"Col correct: {torch.allclose(out[:N], ref_col, atol=1e-2)}")

# Expected on T4:
# Row sum: ~250–300 GB/s (close to the T4's 320 GB/s theoretical peak)
# Col sum: ~30–60 GB/s
# Speedup: 5–10×
#
# The two kernels do exactly the same reduction computation.
# The 5–10× gap comes entirely from one number: stride.
# This is the most important experiment in GPU programming.
Reading the bandwidth numbers

The T4's theoretical peak memory bandwidth is 320 GB/s. Row sum typically achieves 75–90% of this, not 100%, because there is overhead from kernel launch, warp scheduling, and address computation that cannot be eliminated. Column sum achieves 10–20% of peak because the cache-line waste is severe: for a 4096-wide matrix in float16, consecutive threads access addresses 8,192 bytes apart, guaranteeing 32 separate DRAM transactions per warp load. Roughly 31 of every 32 bytes fetched from DRAM are wasted.

Profiling the difference with Nsight Compute

After running both kernels, Nsight Compute can show you exactly what's happening. The metric you want is l1tex__average_t_sectors_per_request_pipe_lsu_mem_global_op_ld.ratio. This measures average DRAM sectors (32 bytes each) fetched per load request. For perfectly coalesced access: 4 sectors (128 bytes = 4 × 32B sectors, one transaction per warp). For completely uncoalesced access: 32 sectors (one sector per thread, no sharing).

ncu --metrics \
  l1tex__average_t_sectors_per_request_pipe_lsu_mem_global_op_ld.ratio,\
  l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum \
  python your_benchmark.py

# Row sum:  sectors/request ≈ 4    (1 transaction per 32 threads, 128 bytes)
# Col sum:  sectors/request ≈ 28–32 (each thread's own transaction)
# This ratio is the coalescing quality metric. 4 = perfect. 32 = worst case.

The sector ratio is a more precise measure than effective bandwidth alone because it controls for how much actual computation the kernel does. A kernel with a ratio of 4 is using hardware-coalesced access regardless of anything else. A kernel with a ratio of 28 is wasting 7 of every 8 bytes it fetches from DRAM.

Free practice

Hands-on exercises — Google Colab T4 or Kaggle free GPU

Setup: Triton 2.x ships pre-installed with PyTorch 2.x on Colab. Verify: import triton; print(triton.__version__). If you need to install it: !pip install triton.

Exercise 1: Run the coalesced vs strided benchmark and vary the stride. Copy the row_sum and col_sum kernels above. Then write a third kernel that reads with a configurable stride: offs = tl.arange(0, BLOCK) * STRIDE. Benchmark with STRIDE = 1, 2, 4, 8, 16, 32. At what stride does the bandwidth collapse? The answer reveals the cache-line size in elements. Compute: if DRAM cache lines are 128 bytes and elements are float16 (2 bytes), the throughput should start degrading at stride = 64. Does it?

Exercise 2: Write the vector addition kernel from scratch. Don't look at the Triton tutorial. From memory: decorated with @triton.jit, one program per block of N elements, uses tl.program_id(0), tl.arange, a mask for out-of-bounds handling, tl.load, the addition, and tl.store. After you've written it, verify against PyTorch's a + b, then read the official Triton Tutorial 01 and compare your implementation. The four patterns — program_id, arange, mask, load/store — appear in every Triton kernel you will ever write.

Exercise 3: Measure your kernel's register usage and check for spills. After any Triton kernel compiles, print kernel.n_regs and kernel.n_spills. The T4 has 65,536 registers per SM and a maximum of 1,024 threads per SM, giving 64 registers per thread at full occupancy. If n_regs > 64, you're at reduced occupancy. If n_spills > 0, some of those registers are actually VRAM accesses. Deliberately force spills by creating a kernel that sums a very large number of independent accumulators. Observe the change in n_regs and n_spills, then measure the throughput degradation.

Exercise 4: Read Triton Tutorial 02 (Fused Softmax) at triton-lang.org — but only after attempting this: write a softmax kernel yourself. Don't worry about the online softmax algorithm yet (that's Lesson 15). Just do the naive three-pass version: one pass to find the max, one pass to compute exp(x - max) and sum, one pass to divide. Run it. Verify it matches torch.softmax. Measure its bandwidth efficiency. You'll understand exactly what the fused version is improving when you read the tutorial.

What comes next

You now have the four memory spaces fully mapped: global memory (700 cycles, where all tensors live, dominated by coalescing discipline), shared memory (20 cycles, programmer-controlled, dominated by bank conflict avoidance), registers (1 cycle, compiler-managed, watch register pressure), and local memory (700 cycles, VRAM alias, appears only through spills or large per-thread arrays). The access pattern to global memory is the most important fact in kernel optimization. The tiling pattern through shared memory is the universal structure of every fast kernel.

Lesson 03 takes this hardware model into the specific context of PyTorch training. Specifically: during the forward pass, which tensors does autograd save? How much memory do they consume? Why does a single GPT-2 layer use far more memory than the weight tensors alone would suggest? And how does torch.utils.checkpoint.checkpoint() trade the memory cost of saved activations for recompute time — which is, structurally, an application of the same tiling insight from this lesson applied to the time axis instead of the memory axis.

Before moving on: can you state, from memory, why tensor.sum(dim=0) is slower than tensor.sum(dim=1) on a row-major tensor? Can you describe what a bank conflict is and how the +1 padding trick eliminates it? Can you name the Nsight Compute metric that measures coalescing quality, and explain what value of 4 means vs 32? These are the three diagnostics you'll reach for every time a kernel is slower than expected.
