How Dynamo intercepts Python execution

Dynamo's core trick is operating at a level below normal Python code: the CPython bytecode interpreter. When Python compiles your source code, it produces a sequence of bytecode instructions — LOAD_FAST, CALL_FUNCTION, BINARY_ADD, and so on. Normally, CPython's _PyEval_EvalFrameDefault function interprets these one by one.

Dynamo replaces this frame evaluation function with its own custom evaluator. When Dynamo's evaluator encounters PyTorch tensor operations (like torch.mm, F.gelu, or tensor.__add__), it records them as symbolic nodes in the FX graph instead of executing them with real tensor values. The symbolic tensors Dynamo uses carry shape, dtype, and device information but no actual data — they're stand-ins that represent "whatever tensor will be here at runtime."

When Dynamo encounters something it cannot trace symbolically — a call to a function that accesses tensor data at the Python level, a Python if that branches on a tensor value, or an external library call without registered tracing rules — it stops tracing, falls back to executing that code in the normal Python interpreter, then restarts tracing afterward. Each stop-and-restart boundary is a graph break.

What an FX graph is and why it matters

An FX (Functional Exchange) graph is PyTorch's standard intermediate representation for compiled computations. It's a directed acyclic graph where each node represents a single operation (an aten op, a function call, or a placeholder for inputs/outputs) and edges represent tensor data flowing between operations.

FX graphs are pure functional: no in-place operations, no Python side effects, no loops or conditionals. If your model has a for loop that runs a fixed number of iterations over the same operations, Dynamo unrolls the loop — the FX graph has N separate nodes for N loop iterations. This makes the graph larger but enables Inductor to see all the operations and fuse them freely. The trade-off: if N is large (a 128-layer loop), the graph is large and takes longer to compile.

import torch
import torch.fx as fx

# ─── Manually construct and inspect an FX graph ───────────────────────────────
class SimpleBlock(torch.nn.Module):
    def __init__(self):
        super().__init__()
        self.linear = torch.nn.Linear(256, 256)
        self.norm   = torch.nn.LayerNorm(256)

    def forward(self, x):
        return self.norm(torch.nn.functional.gelu(self.linear(x)))

model = SimpleBlock()

# Symbolic trace (Dynamo-style, but using the simpler fx.symbolic_trace)
traced: fx.GraphModule = fx.symbolic_trace(model)
print(traced.graph)
# output:
# graph():
#   %x : [#users=1] = placeholder[target=x]
#   %linear_weight : [#users=1] = get_attr[target=linear.weight]
#   %linear_bias : [#users=1] = get_attr[target=linear.bias]
#   %linear : [#users=1] = call_function[aten.addmm](linear_bias, x, linear_weight.T)
#   %gelu : [#users=1] = call_function[aten.gelu](linear)
#   %norm : [#users=1] = call_module[target=norm](gelu)
#   return norm

# Access individual nodes:
for node in traced.graph.nodes:
    print(f"{node.op:15} {node.name:20} args={node.args}")
# opcode         name                 args
# placeholder    x                    ()
# get_attr       linear_weight        ()
# get_attr       linear_bias          ()
# call_function  linear               (linear_bias, x, linear_weight_t)
# call_function  gelu                 (linear,)
# call_module    norm                 (gelu,)
# output         output               (norm,)

Graph breaks: the enemy of compilation

A graph break does three bad things simultaneously. First, it prevents fusion across the break — operations before and after the break become separate kernels with a VRAM round-trip between them. Second, it forces a CPU-GPU synchronization: if the break is caused by a data-dependent branch (if tensor.item() > 0), Python must wait for the GPU to compute that value before the Python branch can execute, stalling the CPU-GPU pipeline. Third, the breakpoint itself runs in Python's interpreter, which is slow for anything tight.

The performance cost is unpredictable: a single graph break in the middle of an otherwise well-compiled 100-op model can halve the speedup because it splits the model into two 50-op subgraphs, each missing the fusion opportunities that the other half's operations would have enabled.

The critical insight: a model that compiles with zero graph breaks is strictly better than one with 10 breaks, even if all the compiled subgraphs are individually well-optimized. The fusion opportunity exists only within a single compiled region. Write for zero breaks as a first-class engineering goal, not an afterthought.

A catalog of graph break causes with fixes

Break cause 1
Data-dependent control flow

Any if, while, or for statement whose condition depends on a tensor value forces Dynamo to evaluate the condition eagerly and break the graph at that point.

Fix: replace with tensor operations — torch.where, torch.clamp, masked_fill, or restructure to remove the condition entirely.
# ✗ BREAK: condition depends on tensor data
def fn_break(x):
    if x.sum() > 0:         # .sum() is a tensor; Python can't trace through this
        return x * 2
    return x * -1

# ✓ FIX: replace with torch.where — fully traceable
def fn_clean(x):
    scale = torch.where(x.sum() > 0, torch.tensor(2.0), torch.tensor(-1.0))
    return x * scale
# Both branches are now expressed as tensor ops, not Python branches.
# Dynamo traces the entire thing into a single graph.
Break cause 2
tensor.item() inside compiled region

.item() extracts a scalar Python number from a tensor, requiring a GPU-CPU data transfer. This is a data dependency that forces immediate evaluation and breaks the graph.

Fix: move .item() outside the compiled region, into the training loop. The compiled function should return tensors; the calling code can call .item() on them.
# ✗ BREAK: .item() inside forward forces sync
def forward_break(x, threshold):
    y = x * 2
    if y.max().item() > threshold:   # breaks graph AND forces GPU sync
        y = torch.clamp(y, max=threshold)
    return y

# ✓ FIX: keep it tensorial inside, extract scalars outside
def forward_clean(x, threshold):
    y = x * 2
    return torch.clamp(y, max=threshold)  # threshold is a Python float — fine

# In training loop:
# loss, max_val = model(x)    # model returns extra diagnostic tensors
# if max_val.item() > thresh: # .item() called outside compiled region
#     do_something()
Break cause 3
Python print(), logging, and side effects

print(), logging.info(), and any other Python side-effect call inside a compiled function cannot be represented in the FX graph and triggers a break.

Fix: remove debug prints from forward(). Use torch.profiler.record_function() for profiling annotations, which Dynamo knows how to handle. Or wrap the debug section in torch.compiler.disable().
# ✗ BREAK: print inside compiled function
def forward_noisy(x):
    y = x @ self.weight
    print(f"y shape: {y.shape}")   # graph break
    return torch.nn.functional.gelu(y)

# ✓ FIX option 1: remove the print
def forward_clean(x):
    y = x @ self.weight
    return torch.nn.functional.gelu(y)

# ✓ FIX option 2: use record_function for profiling (Dynamo-safe)
def forward_profiled(x):
    with torch.profiler.record_function("linear"):   # Dynamo handles this
        y = x @ self.weight
    return torch.nn.functional.gelu(y)

# ✓ FIX option 3: disable compilation for just that section
def forward_debug(x):
    y = x @ self.weight
    with torch.compiler.disable():   # this block runs in eager
        print(f"y shape: {y.shape}")
    return torch.nn.functional.gelu(y)
Break cause 4
Custom autograd.Function without tracing rules

A torch.autograd.Function subclass with a custom forward() written in Python or C++ is a black box to Dynamo. It cannot trace through the function body, so it breaks the graph at each call.

Fix: register the operation as a custom operator with torch.library so Dynamo knows its input/output signature and can include it in the graph.
# ✗ BREAK: autograd.Function is opaque to Dynamo
class MyActivation(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x):
        ctx.save_for_backward(x)
        return x.clamp(min=0) * (x > 1).float()

    @staticmethod
    def backward(ctx, grad):
        x, = ctx.saved_tensors
        return grad * (x > 0).float() * (x > 1).float()

# ✓ FIX: register as a torch library operator
# Dynamo can now trace through it as an opaque node with known I/O schema
import torch.library

@torch.library.custom_op("mylib::my_activation", mutates_args=())
def my_activation(x: torch.Tensor) -> torch.Tensor:
    return x.clamp(min=0) * (x > 1).float()

@my_activation.register_fake
def _(x):
    return torch.empty_like(x)  # tells Dynamo the output shape
# Now my_activation appears as a single node in the FX graph,
# Dynamo doesn't try to trace into it, and it compiles cleanly.
Break cause 5
Iterating over tensor dimensions with Python loops

A for i in range(tensor.shape[0]) loop whose iteration count is a Python integer is fine — Dynamo unrolls it. But for i in range(n) where n comes from tensor.shape at trace time can cause issues if shapes are dynamic.

Fix: use tensor operations instead of Python loops (torch.stack, vmap, torch.unbind with vectorized ops). If the loop is over a fixed dimension known at compile time, Dynamo will unroll it safely.
PatternDynamo behaviorFix
for i in range(32): Unrolls (fine) None needed. 32 nodes in graph.
for i in range(x.shape[0]): Unrolls if static shape Use dynamic=True if shape varies.
if x.dtype == torch.float16: Specializes at trace time (fine) None needed. Recompiles if dtype changes.
if x.requires_grad: Specializes (fine) None needed.
import numpy; numpy.something(x) Graph break Use torch.from_numpy() outside compiled region.
x[0] = value (in-place index) Graph break (aliasing) Use torch.index_put() or reconstruct the tensor.

Finding breaks with torch._dynamo.explain()

The most important tool for graph break diagnosis is torch._dynamo.explain(). It runs the tracing without executing the compiled code and returns a detailed report: exactly where each break occurred, what Python expression caused it, and how the function was split into subgraphs.

import torch

# ─── Diagnose a real model ─────────────────────────────────────────────────────
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained("gpt2")
model.config.use_cache = False  # important: KV cache causes breaks in training mode
x = torch.randint(0, 50257, (1, 64))

explanation = torch._dynamo.explain(model)(x)
print(f"Number of graphs: {explanation.graph_count}")
print(f"Number of breaks: {explanation.break_reasons.__len__()}")
for i, reason in enumerate(explanation.break_reasons):
    print(f"\nBreak {i+1}:")
    print(f"  Reason: {reason.reason}")
    print(f"  File:   {reason.user_stack[-1].filename}:{reason.user_stack[-1].lineno}")

# ─── Verbose logging to stderr for every break in real time ──────────────────
torch._logging.set_logs(graph_breaks=True)
compiled = torch.compile(model)
compiled(x)  # first call: logs every break as it's encountered
torch._logging.set_logs(graph_breaks=False)  # turn off after diagnosis

# ─── Count total graph breaks in a training step ─────────────────────────────
# A clean model should show 0 or 1 graphs total in explain().graph_count.
# More than 3 graphs is a signal to investigate.
# GPT-2 from transformers typically has 1–3 breaks (config checks, etc.)
# A custom poorly-written model might have 20+ breaks.

Dynamic shapes: compiling variable-length inputs

By default, Dynamo compiles a separate kernel for each unique set of input shapes. If you feed sequences of lengths 100, 101, 102, ..., Dynamo compiles 100 different versions of your model — one per length. On the first encounter of each new shape, there's a compilation pause. For inference services with highly variable input lengths, this can make the compiled version slower than eager in the first hours of operation due to constant recompilation.

dynamic=True tells Dynamo to trace with symbolic shapes — placeholders that represent "some integer N" rather than a specific value. The resulting compiled kernel works for any N, no recompilation needed. The cost is slightly less optimal code: some optimizations that work for a specific shape (like assuming N is a multiple of 64 for Tensor Core alignment) can't be applied when N is unknown.

# ─── Static compilation: one kernel per shape ─────────────────────────────────
model_static = torch.compile(model)
# Sequences of different lengths → separate compilations
for length in [64, 128, 256, 512]:
    x = torch.randint(0, 50257, (1, length))
    model_static(x)  # triggers 4 separate compilations!

# ─── Dynamic compilation: one kernel for all shapes ──────────────────────────
model_dynamic = torch.compile(model, dynamic=True)
for length in [64, 128, 256, 512]:
    x = torch.randint(0, 50257, (1, length))
    model_dynamic(x)  # compiles once on the first call, reuses for all shapes

# ─── Inspecting what gets specialized vs symbolized ──────────────────────────
# With dynamic=True, Dynamo treats tensor dimensions as symbolic unless
# it can prove they're always the same. You can see which dims are symbolic:
torch._dynamo.config.verbose = True  # logs specialization decisions
model_dynamic(torch.randint(0, 50257, (1, 100)))
# Look for lines like: "Specializing s0 = 1 (batch dim)" and
# "Treating s1 = 100 as dynamic (sequence dim)"
When to use dynamic=True

Use dynamic=True for inference services where input lengths vary. Use static (default) for training where batch size and sequence length are fixed — you'll compile once and reuse thousands of times, so the ~5% runtime penalty from dynamic shapes isn't worth it. For training with gradient accumulation where effective batch sizes change, consider static mode with explicit shape bucketing: round all sequences to the nearest multiple of 64 before batching, which limits the number of static compilations to a small fixed set.

Recompilation: when and why it happens

Even without dynamic=True, Dynamo caches compiled graphs and only recompiles when necessary. Understanding the recompilation triggers lets you avoid accidentally causing constant recompilations in your training loop.

Recompilation is triggered by: new input shapes (a different batch size or sequence length than any previously seen), new input dtypes (BF16 input after an FP16 input), changes to Python closures that the compiled function captures (e.g., a learning rate variable that Dynamo "baked in" to the compiled graph), and changes to module parameters that affect the graph structure (rare — most parameter changes are fine because they flow as inputs).

# ─── Detect recompilation events ─────────────────────────────────────────────
torch._dynamo.config.verbose = True

compiled = torch.compile(model)

compiled(torch.randint(0, 50257, (1, 64)))   # compile #1
compiled(torch.randint(0, 50257, (1, 64)))   # reuse #1 (same shape)
compiled(torch.randint(0, 50257, (1, 128)))  # compile #2 (new length)
compiled(torch.randint(0, 50257, (4, 64)))   # compile #3 (new batch size)
compiled(torch.randint(0, 50257, (1, 64)))   # reuse #1 (seen before)

# In the verbose output, watch for:
# "Compiling function forward, code obj ..." → new compilation
# "Cache hit ..." → reuse without recompile
#
# Best practice for training loops with fixed shapes:
# 1. Warm up with the exact shapes you'll use during training
# 2. Verify compilation happens only once per shape
# 3. Keep the same batch size and sequence length throughout training

# ─── Reset the compiled cache if needed ──────────────────────────────────────
torch._dynamo.reset()  # clears all compiled graphs from memory cache
# Use this if you're testing multiple compile modes and want a clean start.
# The on-disk cache (~/.cache/torch_inductor) persists across reset() calls.

Custom operators and the torch.library escape hatch

If you have custom CUDA kernels or C extensions that you've wrapped as torch.autograd.Function, they break the graph at every call. The right solution is to register them as proper custom operators using torch.library, which gives Dynamo a formal schema — input types, output shapes, whether the op mutates its inputs — that lets it include the op in the graph as an opaque node without needing to trace through its internals.

# ─── Registering a custom CUDA kernel as a torch.library operator ─────────────
import torch.library

# Step 1: declare the op schema
@torch.library.custom_op("mylib::fast_gelu", mutates_args=())
def fast_gelu(x: torch.Tensor) -> torch.Tensor:
    # This is the CPU fallback / reference implementation
    return torch.nn.functional.gelu(x)

# Step 2: register a "fake" (abstract) implementation that tells Dynamo
# the output shape/dtype without running the actual computation
@fast_gelu.register_fake
def _(x):
    return torch.empty_like(x)

# Step 3 (optional): register the CUDA implementation
# torch.library.impl("mylib::fast_gelu", "cuda")(my_cuda_kernel_wrapper)

# Step 4: register backward
torch.library.register_autograd(
    "mylib::fast_gelu",
    lambda ctx, grad: grad * gelu_derivative(ctx.saved_tensors[0]),
    setup_context=lambda ctx, inputs, output: ctx.save_for_backward(inputs[0])
)

# Now fast_gelu compiles cleanly:
# - Dynamo includes it as a node in the FX graph
# - AOTAutograd knows how to differentiate through it (via register_autograd)
# - Inductor treats it as an opaque call to a custom CUDA kernel
model_with_custom_op = torch.compile(model_using_fast_gelu)

What comes next

You now understand Dynamo at the level where you can diagnose and fix any graph break. The explain() tool tells you what broke. The catalog of break causes tells you how to fix it. Dynamic shapes handles the variable-length problem. Custom operators handle the custom CUDA kernel problem.

Lesson 09 moves deeper into the pipeline: AOTAutograd. This is the stage that most engineers never look at, which is why they can't diagnose compile regressions. AOTAutograd is where the backward pass is traced symbolically, where the min-cut save/recompute decision happens, and where the joint forward-backward graph is produced. Understanding it explains why some models compile faster than others and why the activation memory footprint of a compiled model can differ from its eager footprint.
