Why you need to measure instead of calculate

Lessons 01 through 05 gave you the formulas. You can now compute the theoretical memory budget for any model, any batch size, any sequence length. You can predict activation memory from the saved-tensor list, KV cache size from the architecture config, optimizer state from the parameter count. These calculations are essential — they tell you whether something should fit before you try it.

But calculations tell you what ought to be true. Measurement tells you what actually is true. And the gap between those two things is where most memory bugs live.

Real training code allocates memory in places the formulas don't account for: logging that materializes tensors, callbacks that store activations, gradient accumulation that doubles the grad buffer, custom layers that save extra intermediates, optimizer state in unexpected dtypes. A single line of losses.append(loss) instead of losses.append(loss.item()) will grow your memory linearly with training steps — and the formula will not predict it.

This lesson is about the measurement tools. There are five of them, at increasing depth and increasing overhead. Knowing which one to reach for — and how to read its output — is what separates engineers who can diagnose memory issues from those who just reduce batch sizes until things stop crashing.

The profiling tool ladder

The five profiling tools form a ladder of increasing depth. Each level reveals something the level below cannot. Reach for the lowest rung that answers your current question.

~zero
overhead
memory_allocated() / max_memory_allocated()
Single number: current bytes in use or peak since last reset. Use for quick checks: "does memory grow per step?" Takes nanoseconds.
torch.cuda.memory_allocated() / 1e9 # current GB in use
~zero
overhead
memory_summary()
Full stats table: allocated, reserved, active, inactive, num_allocs, num_ooms. Answers "what is the overall memory state?" One call, no setup.
print(torch.cuda.memory_summary())
~10%
overhead
_record_memory_history() + _snapshot()
Full allocation timeline with Python call stacks. Every malloc and free captured with the exact line of code that triggered it. Visualise at pytorch.org/memory_viz. Use when you need to know where a large allocation came from.
torch.cuda.memory._record_memory_history(max_entries=100_000)
~20–50%
overhead
torch.profiler.profile()
Combined CPU time + GPU kernel time + memory in one trace. Shows which ops allocate the most and how long each takes. Open in TensorBoard or Chrome trace viewer. Use when you need both time and memory together.
with torch.profiler.profile(profile_memory=True) as p: ...
~5%
overhead
Nsight Systems (nsys)
Full GPU hardware trace: kernel timelines, memory copies, SM utilization, API calls. Free NVIDIA download. Use when you suspect GPU idle time, kernel launch overhead, or PCIe bottlenecks that Python-level tools cannot see.
nsys profile --trace cuda,nvtx python train.py

The overhead figures matter. Running a full memory snapshot for 10,000 training steps will slow training by 10% — use it for 2–3 steps and turn it off. Running Nsight Systems on a 2-hour training run gives a huge trace file and is impractical — use it on a small reproducible example. The zero-overhead tools can stay in production code as telemetry.

Basic memory queries: allocated, reserved, peak

Start every memory investigation with the three basic queries. They cost nothing and answer the first-order question: is memory behaving as expected?

import torch

# ─── The three essential queries ─────────────────────────────────────────────

# 1. Currently allocated: bytes actually in use by live tensors
allocated = torch.cuda.memory_allocated() / 1024**3

# 2. Currently reserved: bytes held by PyTorch's caching allocator.
# PyTorch doesn't return freed memory to CUDA immediately — it keeps it in
# a pool for fast re-allocation. reserved >= allocated always.
reserved  = torch.cuda.memory_reserved()  / 1024**3

# 3. Peak allocated since last reset: the high-water mark
torch.cuda.reset_peak_memory_stats()    # reset before the operation you care about
# ... run your training step ...
peak = torch.cuda.max_memory_allocated() / 1024**3

print(f"Allocated: {allocated:.2f} GB")
print(f"Reserved:  {reserved:.2f} GB")
print(f"Peak:      {peak:.2f} GB")

# ─── Use reset_peak + measure pattern to isolate any operation ───────────────
def measure_peak(fn, *args, **kwargs):
    torch.cuda.synchronize()
    torch.cuda.reset_peak_memory_stats()
    before = torch.cuda.memory_allocated()
    result = fn(*args, **kwargs)
    torch.cuda.synchronize()
    peak  = torch.cuda.max_memory_allocated()
    after = torch.cuda.memory_allocated()
    print(f"  peak delta:   {(peak-before)/1e9:.3f} GB")
    print(f"  net retained: {(after-before)/1e9:.3f} GB")
    return result

# Example: measure peak memory of a single forward pass
model = ...  # your model
x     = torch.randn(4, 512, device='cuda')
loss  = measure_peak(lambda: model(x, labels=x).loss)

Understanding the allocated vs reserved gap

The gap between reserved and allocated is memory held by PyTorch's caching allocator but not currently used. PyTorch holds this memory because returning it to CUDA and re-requesting it later incurs overhead. When you call torch.cuda.empty_cache(), you release this reserved-but-free memory back to CUDA, shrinking reserved to match allocated.

This matters in two situations. First, when switching between training and inference in the same process: after training ends, reserved memory can be very large (all the activations that were freed). Calling empty_cache() before starting inference releases this pool so inference can use the full GPU. Second, when using multiple processes on the same GPU: one process's reserved memory is invisible to others, and it can look like "memory leak" when it's just the caching allocator holding freed blocks.

empty_cache() does not help OOMs during training

A common mistake: calling torch.cuda.empty_cache() inside the training loop to avoid OOM. This never works. During training, allocated == reserved approximately — there's nothing reserved-but-free to release. The OOM comes from allocated memory exceeding the GPU's capacity, and empty_cache() can only reclaim reserved-but-free memory. If you're OOMing during training, you need to reduce batch size, enable gradient checkpointing, or reduce precision. empty_cache() is for the transition between workloads, not for active training.

Memory snapshots: seeing every allocation

The basic queries tell you a number. The memory snapshot tells you a story: every allocation, in order, with the Python call stack that caused it. This is the tool you reach for when the number is higher than expected and you need to know exactly which line of code is responsible.

import pickle, torch
from transformers import AutoModelForCausalLM

model     = AutoModelForCausalLM.from_pretrained("gpt2", torch_dtype=torch.float16).cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)
model.config.use_cache = False

x = torch.randint(0, 50257, (4, 512), device='cuda')

# ─── Step 1: start recording (captures every alloc + free with stack trace) ──
torch.cuda.memory._record_memory_history(max_entries=100_000)

# ─── Step 2: run exactly the steps you want to profile ───────────────────────
for _ in range(3):   # 2-3 steps is enough; more = larger file
    optimizer.zero_grad(set_to_none=True)
    loss = model(x, labels=x).loss
    loss.backward()
    optimizer.step()

# ─── Step 3: capture and save the snapshot ───────────────────────────────────
snapshot = torch.cuda.memory._snapshot()
with open("memory_snapshot.pickle", "wb") as f:
    pickle.dump(snapshot, f)

# ─── Step 4: stop recording ──────────────────────────────────────────────────
torch.cuda.memory._record_memory_history(enabled=None)

# ─── Step 5: download and view ───────────────────────────────────────────────
# On Colab:
from google.colab import files
files.download("memory_snapshot.pickle")

Reading a memory_viz trace

When you open the pickle at pytorch.org/memory_viz you get an interactive chart. Learning to read it is a skill. Here is what to look for in each region of the trace.

The forward pass plateau

The memory trace should show a rapid rise at the start of the forward pass as activations are allocated layer by layer, then a flat plateau that persists for the entire duration of the forward pass. The height of this plateau above the baseline (model weights + grads) is your activation memory. If you've computed the theoretical activation memory from Lesson 03 and it matches the plateau height, you're done — everything is expected. If the plateau is 2× larger than expected, something is saving extra tensors.

What to look for in the plateau: hover over blocks in the upper portion of the plateau (the activations, above the weight baseline). The tooltip shows the Python stacktrace that allocated each block. You should see calls to attention, linear, layer_norm, gelu — the operations in your model's forward pass. If you see calls to logging functions, evaluation code, or external callbacks in the activation region, something is holding tensors it shouldn't.

The backward descent staircase

During the backward pass, activations are freed as each layer's gradient is computed. This should appear as a staircase descending from the plateau peak down to the baseline. Each step represents one layer's activations being released. The staircase should be clean and regular — roughly equal step heights if your layers are similar sizes.

What goes wrong: a flat section in the staircase where memory doesn't decrease. This means some layer's activations are not being freed after its backward pass. The most common cause is a hook that captured the activation tensor and is preventing garbage collection. Another cause: in-place operations that share storage with saved tensors, preventing the saved tensor from being freed.

Step-over-step growth

Compare the baseline at the start of step 1 vs the start of step 2 vs step 3. The baselines should be identical. If each step starts from a higher baseline than the last, you have a memory leak: something from one step is surviving into the next. The snapshot's allocation events that persist across step boundaries are your leak — their stacktraces identify the source.

torch.profiler: CPU + GPU + memory together

torch.profiler gives you a different view than the snapshot: instead of a memory-only timeline, it shows CPU time, GPU kernel time, and memory together. This is the tool to reach for when you suspect that a memory-intensive operation is also a compute bottleneck, or when you want to find which operators are allocating the most memory.

import torch
from torch.profiler import profile, ProfilerActivity, tensorboard_trace_handler

# ─── Basic usage: profiler as context manager ─────────────────────────────────
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    profile_memory=True,      # capture memory allocations per op
    record_shapes=True,       # record input tensor shapes
    with_stack=True,          # capture Python call stacks (slower)
) as prof:
    for _ in range(3):
        optimizer.zero_grad(set_to_none=True)
        loss = model(x, labels=x).loss
        loss.backward()
        optimizer.step()

# ─── Sort by self_cuda_memory_usage to find biggest allocators ───────────────
print(prof.key_averages().table(
    sort_by="self_cuda_memory_usage",
    row_limit=15
))
# Key columns:
# Name:                the operator (e.g. aten::mm, aten::softmax)
# Self CUDA Mem:       memory allocated DIRECTLY by this op (not sub-ops)
# CUDA Mem:            total memory including ops it calls (total > self)
# Self CUDA Time:      GPU time for this op directly
# CUDA Time Total:     GPU time including all sub-ops
#
# What to look for:
#   High Self CUDA Mem:  this op is the direct allocator of large tensors
#   High CUDA Mem but low Self CUDA Mem: the alloc is in a sub-op it calls
#   Sort by Self for finding root causes. Sort by Total for the full picture.

# ─── Export to TensorBoard for visual inspection ─────────────────────────────
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    profile_memory=True,
    on_trace_ready=tensorboard_trace_handler("./tb_logs"),
    record_shapes=True,
) as prof:
    with torch.profiler.record_function("forward"):
        loss = model(x, labels=x).loss
    with torch.profiler.record_function("backward"):
        loss.backward()

Reading memory_summary() correctly

torch.cuda.memory_summary() prints a table of memory statistics. Most engineers ignore it because the field names are confusing. Here is what each field means and when it tells you something actionable.

print(torch.cuda.memory_summary(device=0, abbreviated=False))

# Output (abbreviated):
# |===========================================================================|
# |                  PyTorch CUDA memory summary                              |
# |===========================================================================|
# Active memory usage:
#   current: 14.32 GiB                 ← live tensors right now
#   peak:    28.61 GiB                 ← OOM if GPU < this
# Reserved memory usage (caching allocator):
#   current: 29.00 GiB                 ← holds freed blocks for re-use
#   peak:    29.00 GiB
# Active allocs:
#   current: 8412                      ← number of live tensor allocations
#   peak:    12847
# Inactive, non-releasable memory:     ← fragmented blocks too small to reuse
#   current: 280.00 MiB
# Inactive, releasable memory:         ← free blocks that could be given to new allocs
#   current: 14.40 GiB                 ← reserved - active = this
# OOM retries:  0                      ← nonzero means you almost OOMed
# Oversize allocations: 0              ← allocs that bypass the caching allocator
#                                         (very large, goes directly to cudaMalloc)

# Key interpretations:
# 1. If reserved >> active: lots of freed but not-returned memory.
#    Normal during training. Reduce with empty_cache() between workloads.
# 2. If "Inactive, non-releasable" is large: memory fragmentation.
#    Blocks are too small to serve new large allocations. Restart or
#    reduce the number of small temporary tensors.
# 3. If "OOM retries" > 0: PyTorch tried and failed to allocate, then
#    triggered GC and retried. You're operating near the memory limit.
# 4. If "Oversize allocations" > 0: some tensors are so large they bypass
#    the pool entirely. Check your maximum tensor sizes — an unexpectedly
#    large oversize alloc is often a bug (wrong shape, wrong batch size).

The three common training memory leaks

Memory leaks in training loops are not like C memory leaks where a pointer is lost. In Python + PyTorch, leaks happen when objects that reference tensors are kept alive longer than needed — typically by being stored in a Python data structure that outlives the training step. Three patterns account for the vast majority of training loop memory leaks.

Leak 1: storing loss tensors instead of scalars

The most common leak. A tensor is not just a number — it's an object that carries a reference to the entire computation graph that produced it. When you append a loss tensor to a list, you keep the entire backward graph alive in memory for as long as the list lives.

# ─── LEAK: appending tensor keeps the compute graph alive ────────────────────
losses_bad = []
for step in range(100):
    loss = model(x, labels=x).loss
    losses_bad.append(loss)         # BUG: entire forward graph kept alive
    loss.backward()
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
    print(torch.cuda.memory_allocated() / 1e9)  # grows each step!

# ─── FIX: extract scalar first, then append ──────────────────────────────────
losses_good = []
for step in range(100):
    loss = model(x, labels=x).loss
    losses_good.append(loss.item())   # OK: .item() detaches from graph, returns Python float
    loss.backward()
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
    print(torch.cuda.memory_allocated() / 1e9)  # stable after step 1

# Other variants of the same bug:
#   metric_tracker.update(loss)          # if tracker stores the tensor
#   wandb.log({"loss": loss})            # call loss.item() before logging
#   loss_history[step] = loss            # should be loss.item()
#   if loss > best_loss: best_loss = loss # stores tensor reference

Leak 2: gradients not properly freed

Gradient tensors are allocated during .backward() and need to be freed before the next step. optimizer.zero_grad() zeros the gradients but by default keeps the gradient tensors allocated (just fills them with zeros). set_to_none=True actually frees them.

# ─── SLOW: zero_grad() fills grad tensors with zeros but keeps them in memory
optimizer.zero_grad()                    # grad tensors stay allocated (just zeroed)

# ─── FAST AND CORRECT: set_to_none actually frees grad tensor memory
optimizer.zero_grad(set_to_none=True)   # grad tensors freed, set to None

# The difference matters when:
# 1. You have large models (freeing grad tensors = freeing ~16 GB for 8B params)
# 2. You're doing gradient accumulation (accumulated grads can be 2× normal)
# 3. You're running eval between steps (no grads needed during eval)

# ─── Gradient accumulation with proper memory management ─────────────────────
accum_steps = 4
for step, batch in enumerate(dataloader):
    with torch.autocast('cuda', dtype=torch.bfloat16):
        loss = model(**batch).loss / accum_steps  # scale down before backward
    loss.backward()  # accumulates into .grad

    if (step + 1) % accum_steps == 0:
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)  # free grads after each update

Leak 3: hooks that capture activation tensors

Forward and backward hooks are powerful tools for debugging and feature extraction, but they're a common source of leaks because they run inside the autograd context and have easy access to tensors that are still connected to the computation graph.

# ─── LEAK: storing the raw output tensor keeps the graph alive ───────────────
captured_outputs = []

def leaky_hook(module, input, output):
    captured_outputs.append(output)         # BUG: output still in graph

handle = model.transformer.h[0].register_forward_hook(leaky_hook)

# ─── FIX: detach AND move to CPU ─────────────────────────────────────────────
captured_safe = []

def safe_hook(module, input, output):
    captured_safe.append(output.detach().cpu())  # detach cuts graph; .cpu() frees GPU mem

handle2 = model.transformer.h[0].register_forward_hook(safe_hook)

# ─── Also: remove hooks when done ────────────────────────────────────────────
handle.remove()   # important! forgotten hooks keep firing forever
handle2.remove()

# ─── Diagnosis: detect hooks on a model ──────────────────────────────────────
for name, module in model.named_modules():
    if module._forward_hooks:
        print(f"Hook on {name}: {module._forward_hooks}")
    if module._backward_hooks:
        print(f"Backward hook on {name}: {module._backward_hooks}")

Nsight Systems: the hardware view

Nsight Systems is NVIDIA's free system-level profiler. It runs outside Python, traces the GPU at the hardware level, and shows you things no Python-level tool can: individual kernel launch timelines, memory copy bandwidth utilization, SM occupancy over time, CPU-GPU synchronization stalls, and the exact duration of each CUDA kernel in microseconds.

Use Nsight Systems when you suspect:

GPU idle time — white gaps in the GPU timeline where compute stops, usually caused by CPU-GPU synchronization or PCIe transfers that the Python profiler hides behind aggregated op times.

Memory copy bottlenecks — data transfers between CPU and GPU that are blocking compute, visible as MemcpyH2D or MemcpyD2H operations on the timeline.

Kernel launch overhead — for small operations that launch many tiny kernels, the overhead of launching each kernel (5–15 μs per launch) can exceed the computation time. This is visible as many short kernel bars with large gaps between them.

# ─── Basic Nsight Systems usage ───────────────────────────────────────────────
# Download free from developer.nvidia.com/nsight-systems
# Run from command line (not inside Python):

# nsys profile \
#   --trace cuda,nvtx,osrt \         # trace CUDA kernels, NVTX markers, OS calls
#   --output profile_output \        # output file prefix
#   --force-overwrite true \
#   python train_script.py
#
# Opens the .nsys-rep file in Nsight Systems GUI (also free, same download).
# The GUI shows:
#   Top row: CPU thread timeline (Python, CUDA API calls)
#   Middle:  GPU CUDA stream timelines (kernel execution bars)
#   Bottom:  Memory transfer bandwidth
#
# What to look for:
#   White gaps in GPU rows = GPU idle = CPU is the bottleneck
#   Short bar + long gap = kernel launch overhead dominates (use torch.compile)
#   Large MemcpyH2D = data loading is blocking training

# ─── Add NVTX markers to your code for readable Nsight traces ────────────────
import torch.cuda.nvtx as nvtx

for step, batch in enumerate(dataloader):
    with nvtx.range(f"step_{step}"):
        with nvtx.range("data_to_gpu"):
            batch = {k: v.cuda() for k, v in batch.items()}
        with nvtx.range("forward"):
            loss = model(**batch).loss
        with nvtx.range("backward"):
            loss.backward()
        with nvtx.range("optimizer"):
            optimizer.step()
            optimizer.zero_grad(set_to_none=True)
# NVTX markers appear as colored bars in Nsight's GPU timeline,
# making it trivial to identify which region each kernel belongs to.
