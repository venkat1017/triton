Why GPU architecture is the prerequisite

Most engineers learn GPU programming backwards. They start with PyTorch, add torch.compile, sprinkle in some Triton, and then hit a wall. Kernels are slower than expected. Profiles don't make sense. A single-character code change causes a 10× slowdown and nobody knows why.

This happens because GPU optimization is not a bag of tricks. It's a set of consequences that follow inevitably from how the hardware works. Once you have the hardware model in your head, every optimization technique becomes obvious — not something you memorize, but something you derive on the spot.

This lesson is that mental model. It's deliberately hardware-first. We're not touching code yet. By the end, you'll know exactly what a Streaming Multiprocessor is, what a warp is, why memory latency numbers matter more than peak FLOP/s, and how to calculate whether your kernel is even capable of being fast before you write a single line.

Prerequisites

You should be comfortable with Python and have run at least one PyTorch model on a GPU. No CUDA or Triton experience required — this lesson builds the foundation that makes those topics learnable.

The Streaming Multiprocessor

A GPU is not one big compute unit. It's a collection of relatively independent processors called Streaming Multiprocessors (SMs). The A100 has 108 of them. The free Colab T4 has 40.

Each SM is a near-complete processor. It has its own register file, its own L1 cache and shared memory, its own warp schedulers, and its own tensor cores (on Ampere and newer). The SMs share L2 cache and VRAM (HBM), but that's it. Two thread blocks running on two different SMs cannot communicate without going to L2 or VRAM — which costs hundreds of cycles.

The immediate practical implication: if you launch 10 thread blocks on an A100 with 108 SMs, 98 SMs are sitting idle. You've paid for 108 processors and used 10. No compiler optimization touches this. The only fix is to launch enough work to fill the machine.

T4 vs A100

The free Colab T4 has 40 SMs, not 108. All the principles in this lesson apply identically to both GPUs. For Lessons 13–18 (Triton kernels) you'll want A100 access for realistic benchmarking, but T4 is fine for everything conceptual and most micro-benchmarks.

Warps: the unit of execution

Within an SM, threads don't execute independently. They're grouped into warps of 32 threads. All 32 threads in a warp execute the same instruction at the same time on different data. This is the SIMT model — Single Instruction, Multiple Threads.

When the warp scheduler issues an instruction, it issues it to all 32 threads simultaneously. The 32 threads have different register values (they each know their thread index and have their own private registers), but they're executing the same opcode at the same clock cycle.

The consequence is severe if you don't respect it. An if/else that splits a warp in half doesn't give you two branches running in parallel — it gives you two branches running serially, with half the hardware masked during each. This is why data-dependent branching is one of the most expensive things you can do in a kernel.

The practical fix: instead of if/else on tensor values inside a kernel, use torch.where(condition, a, b). The GPU computes both a and b for all threads, then selects. No divergence, full utilization.

The warp is the minimum billing unit

Even if only 1 of the 32 threads in a warp does work (the rest are masked), the warp still occupies a full slot in the warp scheduler. You cannot get sub-warp granularity of resource usage. When you allocate shared memory, it's allocated per-thread-block, not per-active-thread. Always think in multiples of 32.

Latency hiding: why 2048 threads per SM

Each SM has four warp schedulers (on Ampere). Each scheduler can issue one instruction per clock cycle to one warp. So at most 4 warps are issuing instructions at any given cycle.

But the SM can hold up to 64 warps (2048 threads / 32). The rest are "resident" but waiting: either for a memory load to return, or for a data dependency to resolve. This is the GPU's fundamental latency-hiding mechanism. While one warp waits on a 700-cycle VRAM load, 63 other warps can be issuing instructions. By the time the 700 cycles are up, there's a good chance some other warp has done useful work the entire time.

This is the core design philosophy difference between CPU and GPU. CPUs optimize for single-thread latency (massive caches, out-of-order execution, branch prediction to make one thread go as fast as possible). GPUs optimize for throughput (thousands of concurrent threads to hide the latency they can't eliminate). An individual GPU thread is much slower than a CPU thread. The GPU wins on aggregate work done per second.

The memory hierarchy (with exact numbers)

GPU memory is a hierarchy — not just conceptually, but physically. Multiple tiers of different SRAM and DRAM technologies, each with different latency, bandwidth, and capacity tradeoffs imposed by the physics of circuit design. The numbers below are not estimates. Know them cold.

Let's make the numbers visceral. A GPU FMA (fused multiply-add) takes 1 cycle. Accessing a register takes 1 cycle. But going to VRAM takes ~700 cycles. While you're waiting for that VRAM load to return, the compute units could have done 700 multiply-adds.

Shared memory sits in between at ~20 cycles, on-chip, physically inside the SM. The standard technique in every high-performance kernel: load a tile of data from VRAM into shared memory once, do all computation against shared memory, write results back once. This converts 700-cycle accesses to 20-cycle accesses for all the data that gets reused. In Triton, tl.dot() uses this pattern automatically. In Lesson 16 you'll implement it by hand for matmul.

Memory Latency Bandwidth Capacity Scope
Registers ~1 cycle highest 256 KB / SM per-thread, private
Shared Mem / L1 ~20 cycles ~19 TB/s 192 KB / SM per-SM, shared within block
L2 Cache ~200 cycles ~3.1 TB/s 40 MB total all SMs share one L2
HBM / VRAM ~700 cycles 2 TB/s 80 GB off-chip, global
The local memory trap

"Local memory" is a GPU memory space that sounds fast but isn't. It's VRAM, just indexed per-thread automatically by the compiler when you run out of registers. Same 700-cycle latency. It's used for register spills and large thread-local arrays. You won't see it in your Python code — it happens silently. In Lesson 02 we'll detect spills with ncu --metrics l1tex__t_sectors_pipe_lsu_mem_local_op_ld.sum.

Arithmetic intensity and the roofline model

Knowing the memory hierarchy tells you about the hardware. Arithmetic intensity gives you a framework to tell you whether your specific kernel is limited by compute or by memory — and therefore which optimizations are worth pursuing.

Arithmetic intensity (AI) is the ratio of floating-point operations performed to bytes of memory traffic:

Arithmetic Intensity = FLOPs executed / Bytes transferred (VRAM <-> compute units)

The A100 can do 312 TFLOP/s of BF16 computation but only 2 TB/s of VRAM bandwidth. Divide: 156 FLOP/byte. This is the "ridge point" — the arithmetic intensity where the GPU transitions from being memory-bandwidth-limited to compute-limited.

If your kernel's AI is below 156: it's memory-bound. The compute units are sitting idle waiting for data. Adding faster GPUs, bigger matrices, or more clever math doesn't help. The only fixes are: reduce memory traffic (tiling, fusion, quantization), or increase data reuse (keep data in shared memory for multiple computations).

If your kernel's AI is above 156: it's compute-bound. The bandwidth pipe is underutilized; the arithmetic units are the bottleneck. Memory optimizations won't help much here.

Let's work through the two most important real-world examples:

Elementwise operations (GELU, ReLU, add bias): For 1M BF16 elements, a typical elementwise op does 1–8 FLOPs per element. Memory traffic: read 2 MB + write 2 MB = 4 MB. AI = ~4M FLOP / 4 MB = ~1 FLOP/byte. This is 156× below the ridge point on A100. Elementwise ops are almost always memory-bound, regardless of how you implement them. You can't optimize the math — the bottleneck is the VRAM round-trip. The fix: fuse multiple elementwise ops into one kernel. That's exactly what torch.compile does.

Matrix multiplication (4096×4096 × 4096×4096): FLOPs = 2 × 4096³ ≈ 137 GFLOP. Memory traffic (with tiling, so most data comes from SMEM not VRAM): read A + B once from VRAM = ~128 MB. AI ≈ 137 GFLOP / 128 MB ≈ 1070 FLOP/byte. Well above the ridge: compute-bound. This is why matmul is the operation that actually saturates GPU compute.

Why fused kernels are always faster for elementwise ops

An unfused linear → GELU → dropout makes 6 trips to VRAM (read+write per op). A fused kernel does all three in one VRAM round-trip. The arithmetic intensity multiplies by 3, which still doesn't push it above the ridge — but the 3× reduction in memory traffic directly translates to 3× faster wall time. This is the dominant source of torch.compile speedup on transformers. More on this in Lesson 10.

Occupancy

Occupancy is the fraction of an SM's maximum warps that are actually resident and eligible to run. An A100 can hold 64 warps per SM (2048 threads ÷ 32). If your kernel runs 16 warps per SM, occupancy is 25%.

Low occupancy isn't inherently bad. A compute-bound kernel that rarely stalls on memory can run at 25% occupancy and still saturate the ALUs. But for memory-bandwidth-bound kernels — which is most kernels — low occupancy is catastrophic. When a warp stalls on a 700-cycle VRAM load, the scheduler needs other resident warps to fill the idle cycles. No resident warps means the SM goes idle.

Two hardware resources limit occupancy, and they compete with each other:

Shared memory per block. The A100 has 192 KB of shared memory per SM. If your kernel allocates 64 KB of shared memory per thread block, at most 3 blocks can be resident simultaneously (3 × 64 KB = 192 KB). If each block contains 128 threads (4 warps), that's 12 warps out of 64 possible — 19% occupancy.

Registers per thread. The A100 has 65,536 registers per SM. A kernel using 64 registers per thread with 256-thread blocks uses 16,384 registers per block. At most 4 blocks fit (65,536 / 16,384 = 4), giving 32 warps — 50% occupancy. If the compiler decides it needs 128 registers per thread, only 2 blocks fit: 25% occupancy.

How do you detect register spilling? You won't see it in Python. You need the profiler:

ncu --metrics l1tex__t_sectors_pipe_lsu_mem_local_op_ld.sum python my_script.py

A non-zero value means your kernel is using local memory (VRAM) for register spills. We'll instrument this in Lesson 06.

Key numbers to memorize

GPU optimization requires quick mental arithmetic. You need the following numbers to be reflexive — things you can multiply in your head while reading a profile or designing a kernel.

700

156

Derived exercise 1: How many total warps can a T4 run simultaneously? T4: 40 SMs, 32 warps/SM max (1024 threads/SM ÷ 32). Answer: 40 × 32 = 1,280 concurrent warps. If your training loop launches fewer threads than this globally, your T4 is underutilized.

Derived exercise 2: What arithmetic intensity makes a T4 kernel compute-bound? T4: 65 TFLOP/s FP16, 320 GB/s bandwidth. Ridge: 65 × 10¹² / 320 × 10&sup9; = ~203 FLOP/byte. Harder to reach than A100's 156. A kernel at 160 FLOP/byte is compute-bound on A100 but still memory-bound on T4.

BF16 vs FP32 compute

The TFLOP/s numbers above are for Tensor Core BF16. FP32 on regular CUDA cores is roughly 8–16× slower for the same mathematical operation. Any benchmark that doesn't use torch.autocast() or explicitly request Tensor Core paths is running at the much lower number. One of the first things you do after training a model is verify in Nsight Compute that you're actually hitting Tensor Cores, not accidentally running in FP32 on CUDA cores.

Hands-on exercises — Google Colab T4 (free)

Exercise 1: Enumerate your GPU. Before running the code below, write down your guesses: How many SMs does a T4 have? How many max warps per SM? How much VRAM?

import torch

props = torch.cuda.get_device_properties(0)
print(f"GPU: {props.name}")
print(f"SMs: {props.multi_processor_count}")
print(f"Max threads/SM: {props.max_threads_per_multi_processor}")
print(f"Max warps/SM: {props.max_threads_per_multi_processor // 32}")
print(f"Total concurrent warps: {props.multi_processor_count * props.max_threads_per_multi_processor // 32}")
print(f"VRAM: {props.total_memory / 1e9:.1f} GB")
print(f"Shared mem per SM: {props.max_shared_memory_per_multiprocessor / 1024:.0f} KB")

Exercise 2: Calculate the ridge point for your T4. Using the T4's 65 TFLOP/s FP16 and 320 GB/s bandwidth, calculate the arithmetic intensity threshold. Then determine: is a simple y = x * 2.0 on 100M float16 elements compute-bound or memory-bound?

# Arithmetic intensity of y = x * 2.0
N = 100_000_000
flops = N * 1          # one multiply per element
bytes_moved = N * 2 * 2  # float16=2B, read+write
ai = flops / bytes_moved

t4_ridge = 65e12 / 320e9  # ~203 FLOP/byte
print(f"Arithmetic intensity: {ai:.3f} FLOP/byte")
print(f"T4 ridge point: {t4_ridge:.1f} FLOP/byte")
print(f"Memory bound: {ai < t4_ridge}")  # answer: True — deeply memory bound

Exercise 3: Observe the L2 cache vs VRAM speed difference. Benchmark two sums: one on a tensor that fits in L2 (small), one that doesn't (large). The throughput difference shows you the real-world gap between L2 and HBM bandwidth.

from torch.utils import benchmark

# Small: ~4 MB, fits in L2 cache (T4 has 4 MB L2)
x_small = torch.randn(512, 1024, device='cuda', dtype=torch.float16)
# Large: ~256 MB, much larger than L2
x_large = torch.randn(8192, 8192, device='cuda', dtype=torch.float16)

t_s = benchmark.Timer(stmt="x.sum()", globals={"x": x_small}).timeit(500)
t_l = benchmark.Timer(stmt="x.sum()", globals={"x": x_large}).timeit(200)

bw_small = x_small.nbytes / t_s.mean / 1e9
bw_large = x_large.nbytes / t_l.mean / 1e9

print(f"Small tensor effective BW: {bw_small:.1f} GB/s (L2-resident)")
print(f"Large tensor effective BW: {bw_large:.1f} GB/s (VRAM-bound)")
print(f"L2/VRAM ratio: {bw_small/bw_large:.1f}x")

Expected: the small tensor runs at something close to L2 bandwidth (~1+ TB/s theoretical, though Python overhead means you won't see the theoretical max). The large tensor is bounded by the T4's 320 GB/s HBM. The ratio is your empirical measurement of the L2 cache speedup — a real consequence of the memory hierarchy you just learned.
