The autoregressive generation problem

Language models generate text one token at a time. To generate token 512, the model computes attention over all 511 preceding tokens. To generate token 513, it computes attention over 512 tokens. The attention computation at step T processes a sequence of length T — each step is longer than the last.

In attention, the computation for one query token is: scores = Q @ K.T / sqrt(d_head), then output = softmax(scores) @ V. Q is the current token's query. K and V are all tokens' keys and values. To produce these K and V matrices for a sequence of length T, every token must run through the Key and Value projection weights: K = x @ W_K, V = x @ W_V. Naive: recompute all K and V from scratch at every generation step. For a 1024-token sequence at step 1025: compute 1025 key projections and 1025 value projections when you only actually need the one new token's projections.

The insight is obvious once stated: K and V for past tokens never change. Token 1's key and value are the same at step 512 as they were at step 1. Recomputing them is pure waste. Cache them. This is the KV cache: a running store of every past token's K and V tensors, maintained across generation steps so each new step only computes one new pair of K and V vectors, then reads the rest from the cache.

The benefit is enormous: generation cost drops from O(T²) to O(T) total work as the sequence grows. Without caching: generating a 4096-token response requires roughly 4096² /2 = 8 million attention computations. With caching: roughly 4096 × 1 = 4096 new computations, plus reading the cache. The compute savings scale quadratically. The memory cost: you must store the cache. And the cache grows with every token generated.

The KV cache formula, derived from scratch

Let's derive the formula precisely. The KV cache stores, for each attention layer, a K tensor and a V tensor. Their shape is determined by how attention is structured.

In Multi-Head Attention (MHA), each attention head has its own independent K and V projections. If the model has H attention heads and head dimension d_head (where d_head = d_model / H), then the K tensor for one layer has shape [num_tokens, H, d_head] and the V tensor has the same shape. Two tensors per layer. One layer's KV contribution to the cache:

bytes_per_layer = 2 × num_tokens × H × d_head × bytes_per_element

The factor of 2 is one tensor for K, one for V. Across all L layers, and adding a batch dimension (each request in a serving system gets its own KV cache):

KV cache bytes = 2 × L × H_kv × d_head × T × B × bytes_per_element

Where: L = number of layers, H_kv = number of KV heads (explained below), d_head = d_model / H, T = current sequence length (tokens), B = batch size or number of concurrent requests, bytes_per_element = 2 for BF16.

Real numbers: Llama-3-8B at every scale

Let's work through concrete numbers. Llama-3-8B: 32 layers, 32 query heads, 8 KV heads (GQA — we'll explain this shortly), head dimension d_head = 128, BF16 (2 bytes per element).

def kv_cache_gb(layers, kv_heads, d_head, seq_len, batch, bytes_per_elem=2):
    """KV cache size in GB."""
    total_bytes = 2 * layers * kv_heads * d_head * seq_len * batch * bytes_per_elem
    return total_bytes / 1e9

# Llama-3-8B: 32 layers, 8 KV heads (GQA), d_head=128, BF16

Grouped Query Attention: the 4× architectural fix

Standard Multi-Head Attention (MHA) has one Key projection and one Value projection per attention head. If you have 32 heads, you have 32 K matrices and 32 V matrices. For KV caching, this means caching 32 heads' worth of K and V for every token — the cache grows proportionally to the number of heads.

Grouped Query Attention (GQA) breaks this one-to-one correspondence. Multiple query heads share a single Key-Value pair. Llama-3-8B has 32 query heads and 8 KV heads. Four query heads share each KV head. The KV cache only stores 8 heads' worth of K and V — a 4× reduction in cache size.

Multi-Query Attention (MQA) takes this to the extreme: all H query heads share a single KV head. H_kv = 1. Cache reduction = H_q ×. Falcon used this. The quality impact is slightly larger than GQA but the cache is maximally small. GQA is the mainstream choice in 2024 models — it's the sweet spot between memory efficiency and model quality.

d_head is fixed by architecture choice

For Llama-3-8B: d_model = 4096, 32 query heads, so d_head = 4096/32 = 128. With GQA, the 8 KV heads also have d_head = 128 (not d_model/8 = 512 — the head dimension stays the same, there are just fewer heads). This is a critical distinction: GQA reduces H_kv but preserves d_head. Each KV head is the same size as before; there are just fewer of them. The W_K and W_V weight matrices are correspondingly smaller: [d_model, H_kv × d_head] instead of [d_model, H_q × d_head].

Paged attention: the vLLM memory management innovation

The KV cache formula tells you the size of the cache. What it doesn't tell you is how to manage that memory when you're serving hundreds of concurrent requests, each at a different stage of generation, each with a different number of tokens.

The naive approach: allocate max_seq_len × layer × head × d_head × bytes of contiguous VRAM for each request when it arrives. For Llama-3-8B at max_seq_len=4096: 2 × 32 × 8 × 128 × 4096 × 2B = 0.27 GB per request, allocated upfront. If you reserve 200 concurrent request slots: 200 × 0.27 GB = 54 GB, even if most requests are only 50 tokens long. The waste is enormous, and the worst part is the fragmentation: when a 4096-token allocation is freed, it can only be used by another request that needs at most 4096 tokens. Smaller requests can't share the space.

Paged attention, the core innovation in vLLM, borrows the OS virtual memory paging concept. Instead of one large contiguous allocation per request, the KV cache is divided into pages of fixed size (e.g., 16 or 32 tokens each). A request is allocated pages on-demand, one page at a time, as it generates tokens. Multiple requests share pages from the same pool. When a request finishes, its pages are freed immediately and returned to the pool for other requests.

Continuous batching: throughput at scale

Paged attention enables a second optimization that has a larger impact on throughput than page management itself: continuous batching.

In static batching, a serving system waits for all requests in a batch to finish generating before accepting new ones. If your batch has 8 requests and 7 have finished (generated their end-of-sequence token) but 1 is still going, those 7 GPU slots sit idle waiting for the 1 to finish. Real workloads have highly variable sequence lengths — a 5-token completion and a 2000-token story generation in the same batch. Static batching turns this variance into waste.

Continuous batching (also called dynamic batching or in-flight batching): the serving system operates on a per-token basis rather than a per-request basis. At each generation step, it checks if any requests have completed. If so, their pages are freed and new requests can be inserted into the batch to fill those slots. The batch composition changes at every generation step. A 200-request server continuously runs at near-full utilization because empty slots are filled as fast as they open.

The throughput improvement on real workloads is 3–10×. This is the primary reason vLLM is more efficient than naive HuggingFace generate() for serving: it's not the KV cache per se, it's continuous batching enabled by paged memory management. The memory accounting for pages enables the scheduler to know, in real time, whether there's space to start a new request.

KV cache quantization

The KV cache stores values that are used for attention computation. Like weights and activations, they can be quantized to lower precision. INT8 KV cache halves the memory versus BF16. FP8 KV cache halves it again versus INT8. The quality impact is smaller than weight quantization because each KV entry is only read a handful of times — the attention computation just needs rough similarity rankings between Q and K, not exact values.

vLLM supports INT8 and FP8 KV cache quantization in recent versions. The practical effect: at seq=128K with Llama-3-8B, BF16 KV cache is 8.6 GB per request. INT8 reduces this to 4.3 GB. FP8 to 2.15 GB. With FP8 KV cache quantization, a single A100 can serve Llama-3-8B at 128K context with roughly 3 simultaneous users instead of 1 — a significant throughput multiplier for long-context workloads.

Computing KV cache size in code

from transformers import AutoConfig

def kv_cache_gb(model_name_or_config, seq_len, batch=1, dtype_bytes=2):
    """
    Compute KV cache memory in GB from a HuggingFace model config.
    dtype_bytes: 2=BF16/FP16, 1=INT8/FP8
    """
    if isinstance(model_name_or_config, str):
        cfg = AutoConfig.from_pretrained(model_name_or_config)
    else:
        cfg = model_name_or_config

    # Handle both MHA and GQA/MQA architectures
    n_layers   = cfg.num_hidden_layers
    n_q_heads  = cfg.num_attention_heads
    n_kv_heads = getattr(cfg, 'num_key_value_heads', n_q_heads)  # default: MHA
    d_model    = cfg.hidden_size
    d_head     = d_model // n_q_heads  # head dim is fixed by d_model/n_q_heads

    total_bytes = 2 * n_layers * n_kv_heads * d_head * seq_len * batch * dtype_bytes
    gb = total_bytes / 1e9

    return gb, {
        'n_layers': n_layers, 'n_kv_heads': n_kv_heads,
        'd_head': d_head, 'group_factor': n_q_heads // n_kv_heads
    }

# Example: compare models at multiple context lengths
models = {
    "Llama-3-8B":  (32, 32,  8,  128),  # (layers, n_q, n_kv, d_head)
    "Llama-3-70B": (80, 64,  8,  128),
    "Mixtral-8x7B":(32, 32,  8,  128),
    "GPT-4o-like": (96, 64,  8,  128),  # hypothetical
}

print(f"{'Model':<15} {'4K ctx':>8} {'32K ctx':>8} {'128K ctx':>10} {'group factor':>14}")
print("-" * 58)
for name, (L, n_q, n_kv, d_h) in models.items():
    def gb(seq): return 2 * L * n_kv * d_h * seq * 2 / 1e9
    print(f"{name:<15} {gb(4096):>7.2f}G {gb(32768):>7.2f}G {gb(131072):>9.2f}G {n_q//n_kv:>14}x")

# Output:
# Model           4K ctx   32K ctx   128K ctx   group factor
# ----------------------------------------------------------
# Llama-3-8B      0.27G    2.15G     8.59G             4x
# Llama-3-70B     0.66G    5.24G    20.97G             8x
# Mixtral-8x7B    0.27G    2.15G     8.59G             4x
# GPT-4o-like     0.75G    6.00G    24.00G             8x

# Capacity planning: given A100 80G, 16GB for model weights, 4GB overhead:
# Available for KV cache = 60 GB
avail_gb = 60
model_kv_per_req_4k = 0.27
max_concurrent_4k = int(avail_gb / model_kv_per_req_4k)
print(f"\nLlama-3-8B, A100 80G, seq=4096: {max_concurrent_4k} concurrent requests")
# → 222 concurrent requests at 4K context!

model_kv_per_req_128k = 8.59
max_concurrent_128k = int(avail_gb / model_kv_per_req_128k)
print(f"Llama-3-8B, A100 80G, seq=128K:  {max_concurrent_128k} concurrent requests")
# → 6 concurrent requests at 128K context

Inspecting vLLM's KV cache allocation

# Install vLLM (on a Colab A100 or locally):
# pip install vllm

from vllm import LLM, SamplingParams

# vLLM automatically handles KV cache allocation using paged attention
llm = LLM(
    model="gpt2",  # use GPT-2 for free T4
    dtype="float16",
    gpu_memory_utilization=0.9,  # fraction of GPU memory to use for KV cache
)

# Inspect internal cache configuration
cache_config = llm.llm_engine.cache_config
print(f"Block size (tokens per page): {cache_config.block_size}")
print(f"Num GPU blocks: {cache_config.num_gpu_blocks}")
print(f"Total KV cache capacity: {cache_config.num_gpu_blocks * cache_config.block_size} tokens")

# Run some prompts to see paged attention in action
prompts = ["The GPU memory hierarchy consists of"] * 10
outputs = llm.generate(prompts, SamplingParams(max_tokens=50))
for o in outputs:
    print(o.outputs[0].text[:80])
