The golden rule: functional forward passes

Three lessons on the compile pipeline boil down to one engineering principle: write forward passes that are pure functions. A pure function has no in-place mutation, no global state reads or writes, no side effects, and no Python control flow that depends on tensor values at runtime. Given the same inputs, it always produces the same outputs through the same computation path.

Dynamo can trace any pure function completely, producing a single FX graph with no breaks. AOTAutograd can differentiate through it cleanly. Inductor can fuse its operations aggressively. The compiler is essentially a transformer that converts pure functions into optimized GPU code — it works best when the input is already in the form it's designed for.

The anti-patterns below are all violations of purity. Each one introduces a dependency that the compiler cannot represent in a static FX graph: a runtime data value, a Python side effect, a memory aliasing relationship, or an external library call. Understanding exactly why each pattern breaks compilation — not just that it does — lets you recognize variations of these problems and fix them on the spot.

Anti-pattern 1: Python containers used conditionally

Iterating over a Python list or dict of tensors is fine — Dynamo handles it by unrolling the iteration at trace time. The problem appears when you select from a container based on a tensor value, or when the container's contents change between calls in a way that Dynamo doesn't know to recompile for.

# ✗ BREAK: selecting a layer based on a step counter stored as a tensor
class AdaptiveModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers    = nn.ModuleList([nn.Linear(256, 256) for _ in range(4)])
        self.step_count = torch.tensor(0)

    def forward(self, x):
        n = self.step_count.item()              # GPU→CPU: graph break
        layer_idx = n % len(self.layers)        # Python int: fine after break
        return self.layers[layer_idx](x)         # second subgraph starts here

# ✓ FIX A: keep the counter in Python (not a tensor), increment in the training loop
class AdaptiveModelFixed(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers    = nn.ModuleList([nn.Linear(256, 256) for _ in range(4)])
        self._step     = 0                       # Python int, not a tensor

    def set_step(self, n: int):                   # called from training loop, not forward()
        self._step = n
        torch._dynamo.reset()                    # trigger recompile if step changes

    def forward(self, x):
        layer_idx = self._step % len(self.layers)  # Python int: Dynamo specialises on it
        return self.layers[layer_idx](x)

# ✓ FIX B: always run all layers, select with torch.stack + tensor indexing
class AdaptiveModelTensor(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.ModuleList([nn.Linear(256, 256) for _ in range(4)])

    def forward(self, x, layer_idx: torch.Tensor):
        # Run all layers, pick result with differentiable indexing
        outputs = torch.stack([layer(x) for layer in self.layers])  # [4, B, D]
        return outputs[layer_idx]                # tensor indexing: fully traceable

Anti-pattern 2: .item() inside compiled regions

.item() extracts a Python scalar from a tensor. To do this, PyTorch must synchronize the CPU with the GPU — the GPU finishes computing the tensor value, copies a single number to CPU memory, and returns it as a Python float or int. This is a hard synchronization barrier. Dynamo cannot cross it without breaking the graph because the resulting Python scalar is not a tensor and cannot be represented as an FX node.

✗ Anti-pattern
def forward(self, x): y = self.linear(x) scale = y.max().item() # sync! if scale > 10.0: y = y / scale return y
✓ Pattern
def forward(self, x): y = self.linear(x) scale = y.max() # keep it tensorial y = torch.where( scale > 10.0, y / scale.clamp(min=1.0), y ) return y

The important thing to understand is that .item() is not just a graph break — it is also a performance hazard independent of compilation. Every .item() stalls the entire CPU-GPU pipeline: the CPU must wait for the GPU to finish computing the value before Python execution can continue. In a tight training loop, this can add milliseconds of dead time per step. The compiled/non-compiled distinction is secondary to this fundamental cost.

Anti-pattern 3: views that outlive their base

A view shares storage with its base tensor. If you create a view, pass it to a compiled region, and then mutate the base, Dynamo may not be able to track the aliasing relationship correctly. The result is either a graph break, a silent correctness error, or an unnecessary copy inserted by the compiler to be safe.

✗ Anti-pattern — aliasing hazard
x = weights.view(-1) # view of weights y = x + 1 # compiled uses x weights[0] = 99.0 # mutates base! # x now silently changed # Dynamo may not track this
✓ Pattern — explicit copy when needed
x = weights.reshape(-1) # copies if needed y = x + 1 # or: detach from the graph x = weights.detach().view(-1) # clear ownership semantics

The broader principle: in compiled code, ownership must be clear. Each tensor should have exactly one writer. Views that share mutable storage violate this. The Dynamo-safe version uses .reshape() (which copies if the view would share storage through a non-contiguous stride) or makes the immutability explicit with .detach().

Anti-pattern 4: unregistered custom autograd Functions

Covered in Lesson 08: a torch.autograd.Function subclass is opaque to Dynamo. Every call to it breaks the graph. The fix is torch.library.custom_op, which registers a schema that Dynamo can include as a single opaque node without tracing into it. Lesson 08 showed the full registration code; here's the summary of what each piece does and why you need it:

# Three things you must register for a custom op to compile cleanly:

# 1. The schema — tells Dynamo about input/output types and mutation semantics
@torch.library.custom_op("mylib::op", mutates_args=())
def my_op(x: torch.Tensor, scale: float) -> torch.Tensor:
    return x * scale   # reference implementation

# 2. The fake/abstract implementation — tells Dynamo the output shape/dtype
# without running the actual computation (used during tracing)
@my_op.register_fake
def _(x, scale):
    return torch.empty_like(x)  # same shape, same dtype as output

# 3. The autograd rule — tells AOTAutograd how to differentiate through it
@torch.library.register_autograd("mylib::op")
def _(ctx, grad_output, *, x, scale):
    return grad_output * scale, None  # gradient w.r.t. x; no grad for float arg

# Without any of these three: Dynamo inserts a graph break at every call.
# With all three: my_op appears as a single opaque node in the compiled graph.

Anti-pattern 5: in-place mutation of non-leaf tensors

In-place operations on tensors that participate in the computation graph cause one of two problems: they either confuse autograd (which may have saved the old value for the backward pass) or they create aliasing that Dynamo can't represent. The symptoms range from graph breaks to silent gradient errors that are very hard to debug.

✗ In-place on graph tensor
y = self.linear(x) # y is in the graph y.add_(self.bias) # in-place! may corrupt # saved y for backward is now wrong # Dynamo may refuse or produce errors
✓ Out-of-place
y = self.linear(x) y = y + self.bias # new tensor, clean graph # autograd saves the correct y # Dynamo traces cleanly

The exception: in-place operations on leaf tensors (model parameters, like weight updates in the optimizer step) are fine. They happen outside the computation graph and are not traced by Dynamo. param.add_(grad) in an optimizer step is the canonical in-place operation and is perfectly safe.

The other exception: certain in-place ops that Dynamo understands explicitly, such as torch.nn.functional.relu_(x), can sometimes be traced and compiled into fused kernels. But the safe default is out-of-place for anything inside forward().

Patterns that compile cleanly

The positive version of the above is a checklist of what to aim for. Models that exhibit all of these properties compile into a single graph, fuse aggressively, and produce the maximum speedup from torch.compile.

Compiling HuggingFace models correctly

The HuggingFace transformers library ships models that are mostly compile-friendly from version 4.38 onward. A few specific steps are required to get zero or near-zero graph breaks:

from transformers import AutoModelForCausalLM
import torch

# ─── Step 1: load in the right dtype ─────────────────────────────────────────
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    torch_dtype=torch.bfloat16,   # specify explicitly — default is FP32
    device_map="auto",
)

# ─── Step 2: configure for training mode ─────────────────────────────────────
model.config.use_cache = False          # KV cache: breaks graph in training
model.gradient_checkpointing_enable(      # optional: reduces activation memory
    gradient_checkpointing_kwargs={"use_reentrant": False}
)

# ─── Step 3: compile BEFORE the training loop ────────────────────────────────
model = torch.compile(model)
# Don't compile inside the loop. torch.compile wraps the model once.
# All subsequent forward() calls use the compiled graph.

# ─── Step 4: check break count before training ───────────────────────────────
x = torch.randint(0, 32000, (1, 64))
expl = torch._dynamo.explain(model)(x)
print(f"Graph breaks before training: {len(expl.break_reasons)}")
# Target: 0-2 breaks. Llama-3 from transformers ≥4.38 typically gets 0-1.

# ─── Step 5: warmup before timing ────────────────────────────────────────────
optimizer = torch.optim.AdamW(model.parameters(), lr=2e-5)
for _ in range(3):  # warmup: first call compiles, subsequent calls use compiled graph
    optimizer.zero_grad(set_to_none=True)
    with torch.autocast('cuda', torch.bfloat16):
        loss = model(x, labels=x).loss
    loss.backward()
    optimizer.step()
# After warmup: run your actual training loop.
gradient_checkpointing + torch.compile interaction

Gradient checkpointing and torch.compile interact: when checkpointing is enabled, the compiled region inside each checkpoint boundary is smaller (just one layer's forward), reducing the fusion opportunity. For models that fit in memory without checkpointing, disable it for better compile speedup. For models that don't fit, keep checkpointing enabled — the memory saving outweighs the slightly reduced fusion. The key is use_reentrant=False, which is required for checkpointing to work correctly with torch.compile.

Benchmarking compiled models: the warmup discipline

Every benchmark of a compiled model has the same failure mode: measuring the first call, which includes compilation. This single mistake can make compiled models appear 10–100× slower than eager because compilation takes 30–120 seconds while eager executes in milliseconds. The result is a meaningless benchmark that leads to wrong conclusions.

The correct procedure has four steps, each one necessary:

from torch.utils import benchmark

# ─── WRONG: measures compilation time ────────────────────────────────────────
compiled = torch.compile(model)
t = benchmark.Timer(
    stmt="compiled(x, labels=x).loss.backward()",
    globals={"compiled": compiled, "x": x}
).timeit(10)   # WRONG: first call of timeit compiles; time includes compile cost

# ─── RIGHT: full correct benchmark ───────────────────────────────────────────
def step(m):
    m.zero_grad(set_to_none=True)
    with torch.autocast('cuda', torch.bfloat16):
        loss = m(x, labels=x).loss
    loss.backward()
    return loss

# Step 1: compile
compiled = torch.compile(model)

# Step 2: warm up — run at least 2 full steps.
#   Call 1: triggers JIT compilation (slow — minutes for large models)
#   Call 2: uses compiled graph, verifies it's working correctly
#   Call 3: fully warm caches, GPU clocks at boost frequency
for _ in range(3): step(compiled)

# Step 3: synchronize before timing starts
torch.cuda.synchronize()

# Step 4: use benchmark.Timer — it handles synchronization internally
t_compiled = benchmark.Timer(
    stmt="step(compiled)",
    globals={"step": step, "compiled": compiled}
).timeit(50)

# For the eager baseline: also warm up (CUDA driver init, kernel cache)
for _ in range(3): step(model)
t_eager = benchmark.Timer(
    stmt="step(model)",
    globals={"step": step, "model": model}
).timeit(50)

print(benchmark.Compare([t_eager, t_compiled]))
print(f"Speedup: {t_eager.mean / t_compiled.mean:.2f}x")
3 steps
Minimum warmup before timing a compiled model. First compiles, second verifies, third stabilises clocks.
50 iters
Minimum timing iterations for a stable mean. benchmark.Timer uses this for statistical confidence.
same shape
Warmup must use the exact same batch size and sequence length as the benchmark — different shapes trigger recompilation.
autocast on
Benchmark with and without autocast to measure the full picture. Compile speedup and precision speedup (Tensor Cores) compound.

Hands-on exercises — Google Colab T4 (free)

Exercise 1: Take a GPT-2 model and audit it for the six checklist items. Load GPT-2, set use_cache=False, and run torch._dynamo.explain(model)(x). For each graph break found, identify which checklist item it violates. Fix the top 2–3 breaks, re-run explain, and confirm the break count drops. Measure the speedup delta from fixing each break using benchmark.Timer.

Exercise 2: Measure the warmup discipline in action. Compile GPT-2. Measure the time of exactly these calls: call 1 (compilation), call 2 (first compiled), call 3 (second compiled), calls 4–53 (benchmark). Plot call time vs call number. The dramatic drop from call 1 to call 2 is the compilation cost. The stable plateau from call 3 onward is the true compiled performance.

times = []
compiled = torch.compile(model)
for i in range(55):
    torch.cuda.synchronize()
    start = torch.cuda.Event(enable_timing=True)
    end   = torch.cuda.Event(enable_timing=True)
    start.record()
    step(compiled)
    end.record()
    torch.cuda.synchronize()
    times.append(start.elapsed_time(end))
    print(f"call {i:3d}: {times[-1]:8.1f} ms")

Exercise 3: Write a "maximally compile-friendly" GPT-2 block from scratch. Write a single transformer block (attention + FFN + two layer norms) using only PyTorch ops, no in-place mutations, no Python control flow on tensors, no custom autograd functions. Compile it with torch.compile, verify 0 graph breaks with explain(), read the generated Triton to count the number of kernel files. Target: 2–4 kernels (attention matmuls + one fused pointwise kernel for the FFN activations).

Exercise 4: Find the .contiguous() calls in GPT-2 and measure their cost. Search the GPT-2 source for .contiguous(). For each one, measure whether removing it (if safe) reduces the kernel count and improves throughput. Some .contiguous() calls are required by downstream ops (certain CUDA kernels require contiguous input); others are defensive and unnecessary. Learn to distinguish them.
