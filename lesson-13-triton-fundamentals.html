Why Triton instead of CUDA C

CUDA C gives you complete control over every aspect of GPU execution. You specify how many threads to launch, how they map to warps, how they synchronize, how they load and store from shared memory, which shuffle instructions to use for warp-level communication. This control is what makes hand-written CUDA the gold standard for maximum-performance kernels at NVIDIA.

The cost of that control is substantial. A correct, high-performance CUDA matmul is 200–400 lines of C++ that manages bank-conflict-free shared memory layout, double-buffering of data loads, warp-level reduction trees, and register pressure explicitly. Getting all of those right takes domain expertise and time. Getting any one of them wrong costs performance silently.

Triton operates at a higher abstraction level — the block level. Instead of writing operations that each thread in a warp performs on a single element, you write operations that a program instance performs on a block of elements. Triton's compiler handles the mapping from block operations to individual thread instructions, shared memory staging, vectorization, and coalescing. You get hardware-competitive performance while writing code that fits in 30–50 lines.

The tradeoff is real: for maximum performance on extreme cases (the absolute fastest possible matmul), hand-written CUDA with expert-level tuning wins. But for the kernels that matter most in practice — fused activations, custom attention variants, specialized normalization — Triton lets you write them in an afternoon instead of a week, and they run at 80–95% of the theoretical hardware optimum. That is the correct engineering tradeoff.

The Triton programming model

Triton's programming model has three concepts that replace CUDA's thread/block/grid hierarchy:

Programs. A kernel launches as many program instances as the grid specifies. Each program instance is identified by its position in the grid: tl.program_id(axis). Programs are the Triton equivalent of CUDA thread blocks — they execute independently and cannot communicate with each other. The programmer decides how much work each program does.

Blocks. Within a program, operations work on blocks of elements simultaneously. A block is a 1D or multi-dimensional array of values that all execute the same instruction in lockstep. A block of size 256 means 256 elements are loaded, processed, and stored by one program instance. Triton handles the mapping from block operations to physical SIMT execution across warps — you never see individual threads.

Pointers and offsets. Memory in Triton is addressed with pointers and offset arrays. tl.load(ptr + offsets, mask=mask) loads a block of elements from the addresses ptr + offsets[0], ptr + offsets[1], ..., ptr + offsets[BLOCK-1]. The offset array is typically constructed with tl.arange(0, BLOCK). This pattern — pointer arithmetic with a range of offsets — is the fundamental memory access idiom in every Triton kernel you'll ever write.

Anatomy of a kernel: the four building blocks

Every Triton kernel is built from the same four primitives, in the same order. Learning them is learning Triton. The variations between kernels are in what computation happens between load and store — the primitives themselves never change.

PrimitiveWhat it doesExample
tl.program_id(axis) Returns this program instance's index along the given axis. Like CUDA's blockIdx. Always the first line of every kernel. pid = tl.program_id(0)
tl.arange(start, end) Creates a 1D block [start, start+1, ..., end-1]. Used to construct the element-index offsets for this program's block. Size must be a power of 2. tl.arange(0, BLOCK)
tl.load(ptr, mask, other) Loads a block of values from global memory. ptr is a block of addresses. mask suppresses out-of-bounds loads. other is the fill value for masked positions. tl.load(x_ptr + offs, mask)
tl.store(ptr, value, mask) Stores a block of values to global memory. Same masking semantics as load. The final operation in every kernel. tl.store(out_ptr + offs, result, mask)

Between tl.load and tl.store, you operate on the loaded block using standard arithmetic, math functions (tl.exp, tl.log, tl.sqrt, tl.maximum), and reductions (tl.sum, tl.max). These operations apply to every element in the block simultaneously — they're SIMD operations at the block level, not scalar operations per element.

Grids, programs, and how work is divided

When you launch a Triton kernel, you specify the grid — the total number of program instances to launch. For a 1D workload of N elements with BLOCK_SIZE elements per program, the grid has ceil(N / BLOCK_SIZE) programs. Triton's triton.cdiv(N, BLOCK_SIZE) computes this ceiling division.

The grid can be multi-dimensional. For a 2D workload (like a matrix where each program handles one tile), the grid is a 2-tuple: (num_row_tiles, num_col_tiles). Each program then uses both tl.program_id(0) and tl.program_id(1) to locate its position. This is the pattern for 2D tiled operations that you'll use in Lessons 15 and 16.

An important property: programs are completely independent. Two programs executing simultaneously on different SMs cannot communicate, cannot see each other's registers or shared memory, and do not need to synchronize. This is what allows the hardware to schedule programs freely across all SMs — the GPU can run any subset of programs in any order and the result is always correct. Your kernel must be designed so that each program's result is independent of every other program's execution order.

Masking: why block sizes must be powers of 2

Triton requires block sizes to be compile-time constants that are powers of 2: 16, 32, 64, 128, 256, 512, 1024. This constraint exists so the compiler can generate fixed-size vector instructions at compile time rather than variable-length loops at runtime.

But real tensor sizes are rarely exact powers of 2. N=4096? Fine, BLOCK_SIZE=256 divides it evenly. N=4000? Not fine — 4000 / 256 = 15.625. The last program (program 15) would need to handle only 4000 - 15×256 = 160 elements, but its offset array still has 256 entries. Elements 160–255 of the last program are out-of-bounds. Without masking, those loads and stores would access garbage addresses.

The mask is a block of boolean values, one per element. offs < N is evaluated element-wise across the entire offset block, producing a boolean block where True means "this element is within bounds, execute the load/store" and False means "this element is out of bounds, suppress it." For tl.load, suppressed positions return the other value (typically 0.0). For tl.store, suppressed positions are not written.

Your first kernel: vector addition

Vector addition is the "hello world" of Triton. It's the simplest possible kernel: load two elements, add them, store the result. Everything else in Triton is a variation of this structure. Learn this pattern completely — the program_id, the arange, the mask, the load, the computation, the store — and you have the skeleton for every kernel in Lessons 14–18.

import triton
import triton.language as tl
import torch

@triton.jit
def add_kernel(
    x_ptr,                   # pointer to first input vector
    y_ptr,                   # pointer to second input vector
    out_ptr,                 # pointer to output vector
    N,                       # number of elements (Python int at runtime)
    BLOCK_SIZE: tl.constexpr # compile-time constant — must be power of 2
):
    # Step 1: which program am I? (0-indexed within the grid)
    pid = tl.program_id(axis=0)

    # Step 2: compute this program's element offsets
    # arange gives [0, 1, 2, ..., BLOCK_SIZE-1]
    # adding pid * BLOCK_SIZE shifts to this program's range
    offs = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)

    # Step 3: build the mask — True for valid elements, False for OOB
    mask = offs < N

    # Step 4: load from global memory
    # other=0.0 means masked (OOB) positions load as 0
    x = tl.load(x_ptr + offs, mask=mask, other=0.0)
    y = tl.load(y_ptr + offs, mask=mask, other=0.0)

    # Step 5: compute — operates on all BLOCK_SIZE elements simultaneously
    result = x + y

    # Step 6: store to global memory — masked positions are NOT written
    tl.store(out_ptr + offs, result, mask=mask)


def add(x: torch.Tensor, y: torch.Tensor) -> torch.Tensor:
    assert x.shape == y.shape
    assert x.is_cuda and y.is_cuda
    out = torch.empty_like(x)
    N   = x.numel()

    # Grid: ceiling division — enough programs to cover all N elements
    BLOCK_SIZE = 1024
    grid = (triton.cdiv(N, BLOCK_SIZE),)  # 1D grid, one axis

    # Launch: kernel[grid](args...)
    add_kernel[grid](x, y, out, N, BLOCK_SIZE)
    return out


# Verify correctness
x   = torch.randn(100_000, device='cuda', dtype=torch.float16)
y   = torch.randn(100_000, device='cuda', dtype=torch.float16)
out = add(x, y)
print(torch.allclose(out, x + y))  # True

# Benchmark vs PyTorch
from torch.utils import benchmark
t_torch  = benchmark.Timer("x + y",         globals={"x":x,"y":y}).timeit(500)
t_triton = benchmark.Timer("add(x, y)",     globals={"add":add,"x":x,"y":y}).timeit(500)
print(f"PyTorch: {t_torch.mean*1e3:.3f} ms")
print(f"Triton:  {t_triton.mean*1e3:.3f} ms")
# Both should be within ~5% of each other on T4 — we haven't added any special
# optimization yet, just reproduced what PyTorch does at the same level.

tl.constexpr and why it matters

BLOCK_SIZE: tl.constexpr is a type annotation that tells Triton this argument is a compile-time constant. When you launch the kernel with BLOCK_SIZE=1024, Triton compiles a version of the kernel specialized for that exact value. tl.arange(0, BLOCK_SIZE) becomes tl.arange(0, 1024) at compile time — the compiler knows the array size and can generate fixed-size vector instructions rather than variable-length loops.

If you change BLOCK_SIZE to a different value at runtime, Triton compiles a new specialized kernel for that value. This is the Triton equivalent of C++ template specialization: one source, many compiled versions, each optimized for its specific parameters. The first call with each new value of a constexpr argument triggers a compilation; subsequent calls with the same value reuse the compiled kernel from cache.

The constraint: tl.constexpr values must be known at the time the kernel is launched, not at compile time in the Python sense. The value can change between launches. What cannot change is the value during kernel execution — it's constant within a single kernel invocation.

What must be constexpr

Any value used as the size argument to tl.arange() or as an array dimension in tl.zeros(), tl.full(), or similar must be constexpr. This includes all tile dimensions: BLOCK_M, BLOCK_N, BLOCK_K, HEAD_DIM. Values used only in arithmetic (like N, stride) do not need to be constexpr — they're runtime values that can be different for every launch.

Three more kernels: square, scale, fused multiply-add

The best way to internalize the Triton skeleton is to write variations of it until the pattern is automatic. Here are three kernels that each introduce one new concept:

# ─── Kernel 1: elementwise square (y = x²) ────────────────────────────────────
# Introduces: single input, scalar computation
@triton.jit
def square_kernel(x_ptr, out_ptr, N, BLOCK: tl.constexpr):
    pid  = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < N
    x    = tl.load(x_ptr + offs, mask=mask, other=0.0)
    tl.store(out_ptr + offs, x * x, mask=mask)


# ─── Kernel 2: scale by a runtime scalar (y = alpha * x) ──────────────────────
# Introduces: runtime scalar argument (not constexpr — alpha changes every call)
@triton.jit
def scale_kernel(x_ptr, out_ptr, alpha, N, BLOCK: tl.constexpr):
    pid  = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < N
    x    = tl.load(x_ptr + offs, mask=mask, other=0.0)
    tl.store(out_ptr + offs, alpha * x, mask=mask)  # scalar broadcasts over block


# ─── Kernel 3: fused multiply-add (out = a * b + c) ───────────────────────────
# Introduces: three inputs, fused computation — the key Triton advantage.
# Without fusion: 3 kernel launches, 6 VRAM accesses.
# With fusion: 1 kernel launch, 4 VRAM accesses (3 reads + 1 write).
@triton.jit
def fma_kernel(a_ptr, b_ptr, c_ptr, out_ptr, N, BLOCK: tl.constexpr):
    pid  = tl.program_id(0)
    offs = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offs < N
    a    = tl.load(a_ptr + offs, mask=mask, other=0.0)
    b    = tl.load(b_ptr + offs, mask=mask, other=0.0)
    c    = tl.load(c_ptr + offs, mask=mask, other=0.0)
    tl.store(out_ptr + offs, a * b + c, mask=mask)   # all three computed in registers


# ─── Benchmark the FMA kernel against unfused PyTorch ─────────────────────────
N    = 50 * 1024 * 1024   # 50M elements
a    = torch.randn(N, device='cuda', dtype=torch.float16)
b    = torch.randn(N, device='cuda', dtype=torch.float16)
c    = torch.randn(N, device='cuda', dtype=torch.float16)
out  = torch.empty_like(a)
BLOCK = 1024
grid  = (triton.cdiv(N, BLOCK),)

from torch.utils import benchmark

t_unfused = benchmark.Timer(
    stmt="a * b + c", globals={"a":a,"b":b,"c":c}).timeit(200)
t_fused   = benchmark.Timer(
    stmt="fma_kernel[grid](a,b,c,out,N,BLOCK)",
    globals={"fma_kernel":fma_kernel,"grid":grid,"a":a,"b":b,"c":c,"out":out,"N":N,"BLOCK":BLOCK}).timeit(200)

bytes_moved = N * 2 * 4  # 3 reads + 1 write, float16
bw_unfused  = bytes_moved / t_unfused.mean / 1e9
bw_fused    = bytes_moved / t_fused.mean   / 1e9
print(f"Unfused a*b+c: {bw_unfused:.0f} GB/s  (3 kernels, 6 VRAM accesses)")
print(f"Fused FMA:     {bw_fused:.0f} GB/s  (1 kernel, 4 VRAM accesses)")
# Expected on T4: fused ~280 GB/s, unfused ~170-200 GB/s
# The fused version is faster because it eliminates the intermediate
# VRAM writes for (a*b) before adding c.
# Verify correctness:
print(torch.allclose(fma_result := (lambda: (fma_kernel[grid](a,b,c,out,N,BLOCK), out)[1])(), a*b+c, atol=1e-2))

Exercise 2: Handle a non-power-of-2 N explicitly. Write a kernel for y = relu(x) (tl.maximum(x, 0.0)) and test it with N=100,003 (deliberately odd). Verify that the masking handles the partial last block correctly. Then test with N=1 and N=BLOCK_SIZE-1. All should produce correct results.

Exercise 3: Compare different BLOCK_SIZE values. Benchmark the vector addition kernel with BLOCK_SIZE ∈ {64, 128, 256, 512, 1024} on N=100M float16 elements. Plot effective memory bandwidth (GB/s) vs BLOCK_SIZE. Where is the optimum on your T4? (Expected: 256–512 is often optimal; below 128 underutilizes memory transactions, above 1024 may reduce occupancy.)

results = {}
for BS in [64, 128, 256, 512, 1024]:
    grid_bs = (triton.cdiv(N, BS),)
    t = benchmark.Timer(
        stmt="add_kernel[g](x,y,out,N,BS)",
        globals={"add_kernel":add_kernel,"g":grid_bs,"x":x,"y":y,"out":out,"N":N,"BS":BS}
    ).timeit(300)
    bw = N * 2 * 3 / t.mean / 1e9  # 2 reads + 1 write, float16
    results[BS] = bw
    print(f"BLOCK={BS:4d}: {bw:.0f} GB/s")
