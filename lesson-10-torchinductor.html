Inductor's position in the pipeline

TorchInductor is the third and final stage of torch.compile. It receives the joint FX graph from AOTAutograd — a complete, pure-functional DAG describing every operation in both the forward and backward passes — and produces executable GPU kernels. On GPU targets, those kernels are written in Triton. On CPU, they're OpenMP-parallelized C++.

Inductor is where the actual performance gains are produced. Everything before this stage — Dynamo's tracing, AOTAutograd's joint graph — is setup. Inductor is the payoff. It's also the most directly connected to Batch 3 of this course: the Triton kernels Inductor generates look almost identical to the kernels you'll write by hand in Lessons 13–18. Reading Inductor's output is a preview of what you'll be authoring directly.

The three things Inductor does that matter for performance:

1. Operator fusion. Adjacent operations with compatible loop structures are merged into a single kernel. This eliminates intermediate VRAM writes and reduces kernel launches. It's the dominant source of speedup for memory-bound models.

2. Autotuning. Inductor benchmarks multiple configurations (block sizes, number of warps, pipeline stages) for each generated kernel and caches the fastest one. It amortizes this search cost across many training steps.

3. CUDA Graphs (via reduce-overhead mode). The entire sequence of kernel launches for a training step is captured as a single GPU "program." Future steps replay this program with one API call instead of launching each kernel individually, eliminating all kernel launch overhead.

Lowering FX to loop-level IR

The FX graph describes computation at the operation level: "compute matmul of A and B," "apply GeLU to tensor X." Inductor needs to go one level lower — the loop level — before it can generate kernel code. This lowering step turns each operation into an explicit nest of loops over tensor dimensions, with the loop body performing a single scalar computation.

Why lower to loops? Because loops are the unit at which fusion is evaluated. Two operations can be fused if and only if their loops iterate over the same index space in the same order. A pointwise add and a pointwise GeLU both loop over all elements of a tensor in the same order — their loops are compatible and can be merged. A matmul and the pointwise add of its bias loop over the same output elements but in very different ways (the matmul involves an inner K-dimension reduction) — they may or may not fuse depending on how their index spaces align.

Fusion rules: what merges, what doesn't

Inductor's fusion decisions determine how many Triton kernels your compiled model generates. Understanding the rules lets you write models that naturally compile into fewer, larger kernels — and avoid the patterns that defeat fusion without knowing why.

PatternFuses?Why
add → mul → gelu → relu ✓ Yes All pointwise, same iteration space. One kernel.
pointwise op → reduction (sum/mean/max along one dim) ✓ Yes The pointwise op feeds directly into the reduction's inner loop. Inductor tiles both together.
reduction → pointwise op using reduction result ✓ Yes (partial) The reduction result is a scalar or smaller tensor; pointwise op consuming it can be fused into the same kernel as a epilogue.
matmul → bias add ✓ Yes Inductor generates a fused matmul+bias kernel. The bias add is an epilogue of the matmul tile loop.
matmul → matmul ✗ No Both have inner reduction loops; they can't share registers for the intermediate. Two separate kernels.
pointwise op → .view() → pointwise op ✓ Usually If the view doesn't change the underlying storage (stride-compatible), Inductor sees through it and fuses.
pointwise op → .contiguous() → pointwise op ✗ No .contiguous() forces a memory copy — a VRAM write/read that breaks the fusion chain.
ops across graph break boundaries ✗ No Graph breaks produce separate subgraphs; fusion only happens within a single subgraph.
softmax (sum + exp + divide) ✓ Partially Inductor fuses the exp and sum into one reduction kernel, but the three-pass naive softmax becomes a fused two-pass kernel — not quite online softmax (you'll do that in Lesson 15).

The single most important anti-pattern is unnecessary .contiguous() calls. Every .contiguous() in your model's forward pass creates a VRAM copy that breaks the fusion chain at that point. PyTorch sometimes inserts implicit contiguous calls during layout-changing operations — transposes followed by matmuls, or views that change the innermost stride. Check your model for implicit .contiguous() calls by enabling Inductor's trace output and looking for aten.copy_ nodes in the lowered IR.

Code generation: reading the generated Triton

Inductor's GPU backend generates Triton code. Understanding the generated code is directly useful for two reasons: it shows you the optimization level you're getting "for free" from the compiler, and it previews the exact patterns you'll write yourself in Batch 3. The generated Triton is verbose but structurally identical to handwritten Triton.

import torch
torch._inductor.config.debug = True
torch._inductor.config.trace.enabled = True

# Compile a fused function and examine the generated code
def fused_layer(x, W, b):
    h    = x @ W + b
    gate = torch.nn.functional.gelu(h)
    return gate * 0.5

compiled = torch.compile(fused_layer)
compiled(torch.randn(4, 512, device='cuda'),
         torch.randn(512, 256, device='cuda'),
         torch.randn(256,        device='cuda'))

Here is a representative example of what Inductor generates for the fused bias add + GeLU pattern. This is actual Triton code, autogenerated. Compare it to what you'll write in Lesson 15 — the structure is identical:

# ── INDUCTOR-GENERATED TRITON (fused: matmul epilogue + gelu + scale) ────────
import triton
import triton.language as tl

@triton.jit
def triton_poi_fused_add_gelu_mul_0(
    in_ptr0,   # matmul output (x @ W)
    in_ptr1,   # bias b
    out_ptr0,  # final output
    xnumel,    # total elements = 4 * 256
    XBLOCK: tl.constexpr,
):
    xoffset = tl.program_id(0) * XBLOCK
    xindex  = xoffset + tl.arange(0, XBLOCK)
    xmask   = xindex < xnumel

    # Load matmul output (already computed by separate mm kernel)
    tmp0 = tl.load(in_ptr0 + xindex, xmask)
    # Load bias — broadcast: each row uses same bias, stride = output_dim
    tmp1 = tl.load(in_ptr1 + (xindex % 256), xmask)

    # Fused: add + gelu + scale — all in registers, no intermediate VRAM writes
    tmp2  = tmp0 + tmp1                          # bias add
    tmp3  = tmp2 * 0.5
    tmp4  = tmp2 * 0.7071067811865476           # 1/sqrt(2) for gelu erf variant
    tmp5  = tl.extra.cuda.libdevice.erf(tmp4)
    tmp6  = tmp5 + 1.0
    tmp7  = tmp3 * tmp6                          # gelu: 0.5 * x * (1 + erf(x/sqrt2))
    tmp8  = tmp7 * 0.5                          # final scale

    tl.store(out_ptr0 + xindex, tmp8, xmask)

# This is the same as what you'd write by hand in Lesson 13–15.
# One tl.load per input, all computation in registers (tmp0..tmp8),
# one tl.store. Zero unnecessary VRAM round-trips.

Four things to notice in Inductor's generated code that you'll replicate in Batch 3:

1. The grid and XBLOCK pattern. tl.program_id(0) * XBLOCK + tl.arange(0, XBLOCK) is the canonical Triton 1D index pattern. Every elementwise kernel uses this exact structure.

2. The mask. xmask = xindex < xnumel handles out-of-bounds elements when N is not a multiple of XBLOCK. Inductor always generates masked loads and stores.

3. Register chaining. tmp0 through tmp8 live entirely in registers. There are exactly two VRAM accesses: one tl.load per input and one tl.store for the output.

4. Math expansion. GeLU, which looks like a single op in FX, expands to five scalar operations in the Triton IR. Inductor inlines the complete math rather than calling a separate gelu function.

Autotuning: how Inductor picks tile sizes

Generated Triton kernels have configuration parameters — primarily XBLOCK (the number of elements processed per program instance) and num_warps (the number of warps per thread block). The optimal values depend on the tensor size, the GPU architecture, and the specific computation. A block size of 256 may be optimal for one tensor shape and suboptimal for another.

Inductor's autotuner solves this by benchmarking multiple configurations on the first call with each new input shape, then caching the winner. It explores a predetermined set of candidate configurations — for a 1D elementwise kernel, typically XBLOCK ∈ {64, 128, 256, 512, 1024} × num_warps ∈ {4, 8} = 10 combinations. Each combination runs a few warmup iterations and then a timed run. The fastest one is saved to disk in ~/.cache/torch_inductor/ keyed by (op type, input shape, device).

# ─── Control autotuning behavior ─────────────────────────────────────────────
import torch._inductor.config as config

# Default: autotune with a reasonable set of configs
config.max_autotune = False   # default: uses heuristic configs only
config.max_autotune = True    # exhaustive: much slower compile, fastest runtime

# Pre-populate the tuning cache by running on representative inputs once
model_compiled = torch.compile(model, mode="max-autotune")

# Warmup with the shapes you'll use at inference/training time
for _ in range(5):   # 5 calls: 1 compiles + tunes, 4 verify and warm up the kernel
    model_compiled(x)

# After warmup, the tuning results are cached on disk.
# Subsequent runs (even after process restart) skip tuning entirely.
# Cache location: ~/.cache/torch_inductor/

# ─── Inspect what configs Inductor tries ─────────────────────────────────────
config.trace.enabled   = True
config.trace.debug_log = True

model_compiled2 = torch.compile(model)
model_compiled2(x)
# In the debug log, look for:
# "Autotuning kernel triton_poi_fused_... with N configs"
# "Config XBLOCK=256, num_warps=4: 0.123ms"
# "Config XBLOCK=512, num_warps=8: 0.098ms"  ← winner
# "Selected XBLOCK=512, num_warps=8"

CUDA Graphs: eliminating kernel launch overhead

Even after fusion reduces the number of kernels, launching kernels still costs CPU time. Each kernel launch is a CPU syscall that enqueues a command on the GPU's work queue. At 5–15 μs per launch, a 50-kernel training step costs 250–750 μs of CPU overhead — overhead that runs on the critical path even if the GPU would otherwise be idle.

CUDA Graphs eliminate this by capturing the entire sequence of kernel launches for one training step as a single "graph object" on the GPU. The first step runs normally and records the sequence. Every subsequent step replays the recorded graph with a single lightweight API call, bypassing all per-kernel CPU overhead. The gains are largest for small models or small batch sizes where kernel launch overhead is a significant fraction of total step time.

# ─── mode="reduce-overhead" enables CUDA Graphs automatically ─────────────────
model_cg = torch.compile(model, mode="reduce-overhead")

# Requirements for CUDA Graphs to work:
# 1. Fixed input shapes (CUDA Graphs record pointer offsets, not shapes)
# 2. No Python-level control flow that changes the kernel sequence
# 3. No CPU-GPU synchronizations inside the step (e.g., .item() calls)
# 4. No in-place operations on inputs between steps

# Verify CUDA Graphs are active:
torch._inductor.config.triton.cudagraph_trees = True  # enable (default in reduce-overhead)

# ─── Manual CUDA Graph for fine control ──────────────────────────────────────
import torch.cuda

# Warmup (must use the same stream as the capture)
s = torch.cuda.Stream()
with torch.cuda.stream(s):
    for _ in range(3):  # warmup passes — allocate all memory
        y = model(x)

# Capture the graph
g = torch.cuda.CUDAGraph()
with torch.cuda.graph(g):
    static_output = model(static_input)  # static_input is a fixed tensor

# Replay — copies new data into static_input, replays graph, reads from static_output
for batch in dataloader:
    static_input.copy_(batch)  # update the static tensor in place
    g.replay()                  # replay the captured sequence
    result = static_output.clone()  # read result

Debugging Inductor: the trace flags

Inductor has a comprehensive set of debug flags. These are the ones you'll actually use when something compiles but produces wrong results or unexpected performance:

import torch._inductor.config as cfg

# ─── Level 1: see what Inductor is generating ─────────────────────────────────
cfg.debug = True            # write generated Triton to /tmp/torchinductor_*/
cfg.trace.enabled = True    # detailed logs of every compilation decision

# ─── Level 2: see fusion decisions ───────────────────────────────────────────
cfg.trace.debug_log = True   # logs which ops fuse, which don't, and why
# Look for lines like:
# "Fusing: [add, gelu] → triton_poi_fused_add_gelu_0"
# "Cannot fuse: [mm, add] — different loop type"

# ─── Level 3: check for fallbacks to eager ───────────────────────────────────
# If Inductor can't lower an op, it falls back to calling the PyTorch eager
# implementation. This is silent by default but measurable:

fallbacks = []
original_op = torch.ops.aten.mm.default

def track_fallback(*args, **kwargs):
    fallbacks.append('mm')
    return original_op(*args, **kwargs)

# After compilation, count fallbacks — should be zero for a clean model
print(f"Inductor fallbacks: {len(fallbacks)}")

# ─── Level 4: kernel count before and after compilation ──────────────────────
import torch.cuda

def count_kernels(fn, *args):
    torch.cuda.synchronize()
    start = torch.cuda.Event(enable_timing=True)
    end   = torch.cuda.Event(enable_timing=True)
    start.record()
    fn(*args)
    end.record()
    torch.cuda.synchronize()
    # Kernel count is only visible via Nsight; this gives elapsed time instead
    return start.elapsed_time(end)

t_eager    = count_kernels(model, x)
t_compiled = count_kernels(compiled_model, x)
print(f"Eager:    {t_eager:.2f}ms")
print(f"Compiled: {t_compiled:.2f}ms")

Writing models that compile into fewer larger kernels

The practical guidance from understanding Inductor's fusion rules comes down to a small set of model authoring principles. These are not theoretical — they're the difference between a model that compiles into 20 kernels and one that compiles into 80.

Use .view() instead of .contiguous() wherever possible. .view() returns a new tensor with different strides but the same storage — no copy, no VRAM write, fusion continues uninterrupted. .contiguous() forces a VRAM copy. Only use .contiguous() when you actually need contiguous memory (e.g., before passing to a C extension that requires it).

Keep pointwise operations together. A sequence of pointwise ops (GeLU, bias add, dropout, layer scaling) fuses into one kernel. If you split them with a reshape or transposition, each side of the split becomes a separate kernel. Group all elementwise post-processing together in one place in the computation graph.

Avoid .item() and tensor-to-Python conversions inside forward(). These force synchronization and break both graph compilation (Lesson 08) and CUDA Graphs. Move all scalar extractions to after the compiled function returns.

Use in-place operations carefully. Some in-place ops (relu_, add_) can be included in the fusion chain. Others break it by creating aliasing that Inductor can't reason about. When in doubt, prefer out-of-place.

# ─── Model design patterns for compilation efficiency ─────────────────────────

# ✓ GOOD: chain of pointwise ops — one fused kernel
def ffn_good(x, W1, W2, b1, b2):
    h = x @ W1              # matmul kernel
    h = h + b1              # ─┐
    h = torch.nn.functional.gelu(h)   #  ├─ all fused into one pointwise kernel
    h = h @ W2              # matmul kernel
    h = h + b2              # ─┘ fused with h@W2 epilogue
    return h
# Total: 2 matmul kernels + 1 fused pointwise kernel = 3 kernels

# ✗ BAD: unnecessary .contiguous() breaks fusion
def ffn_bad(x, W1, W2, b1, b2):
    h = x @ W1
    h = (h + b1).contiguous()   # VRAM write → new tensor → breaks fusion chain
    h = torch.nn.functional.gelu(h)  # separate kernel now
    h = h @ W2
    h = h + b2
    return h
# Total: 2 matmul kernels + 2 pointwise kernels = 4 kernels (33% more)

# ─── Verify kernel count via torch.compile trace ─────────────────────────────
import os, glob

cfg.debug = True
compiled_good = torch.compile(ffn_good)
compiled_bad  = torch.compile(ffn_bad)

for fn, name in [(compiled_good, "good"), (compiled_bad, "bad")]:
    cache_dir = os.path.expanduser("~/.cache/torch_inductor")
    before    = set(glob.glob(f"{cache_dir}/**/*.py", recursive=True))
    fn(torch.randn(4, 512, device='cuda'),
       torch.randn(512, 256, device='cuda'),
       torch.randn(256, 128, device='cuda'),
       torch.randn(256,        device='cuda'),
       torch.randn(128,        device='cuda'))
    after     = set(glob.glob(f"{cache_dir}/**/*.py", recursive=True))
    print(f"{name}: {len(after - before)} new kernel files generated")

Exercise 1: Count kernel files before and after compilation. Enable torch._inductor.config.debug = True. Compile a function with a clean chain of 4 pointwise operations. Count the .py files generated in ~/.cache/torch_inductor/. Then add a .contiguous() in the middle and recount. Verify the bad version generates one more kernel file — confirming the fusion break.

Exercise 2: Read Inductor's generated Triton and relate it to the source. Write a function that does F.gelu(x + bias). Compile it, find the generated .py file in the tmp directory, and read the Triton kernel. Identify: (a) the program_id + arange + mask pattern, (b) where the x + bias computation happens (a register operation), (c) where the GeLU happens (also a register operation), (d) the single tl.store at the end. This is exactly what you'll write from scratch in Lesson 13.

Exercise 3: Benchmark CUDA Graphs vs standard compile on GPT-2. Compare three modes: eager, torch.compile(mode="default"), and torch.compile(mode="reduce-overhead") on batch size 1. At small batch sizes, reduce-overhead should be significantly faster than default because kernel launch overhead dominates. At batch size 16, the advantage should narrow. Use torch.utils.benchmark.Timer with 100 iterations and proper warmup.

Exercise 4: Autotune and inspect the cache. Compile a large matmul (e.g., torch.compile(lambda A, B: A @ B, mode="max-autotune")) on 1024×1024 matrices. Look in ~/.cache/torch_inductor/ for the tuning log. Find the winning configuration (XBLOCK / num_warps / num_stages). Then change the matrix size to 2048×2048 and recompile — verify it produces a different cached configuration, since optimal tile sizes depend on problem size.

What comes next

You've now completed Batch 2. You understand the full torch.compile stack: Dynamo captures Python bytecode into an FX graph, AOTAutograd traces the backward pass ahead of time and applies min-cut partitioning to minimize saved tensors, Inductor lowers the FX graph to a loop-level IR, fuses adjacent loops into single kernels, generates Triton code, and autotunes the configuration. The speedup is not magic — it's a direct consequence of fewer VRAM round-trips (fusion), fewer kernel launches (fusion + CUDA Graphs), and hardware-optimal tile sizes (autotuning).

Batch 3 begins with Lesson 11 (compile-friendly code patterns) and then moves to the kernels themselves. You've been reading Inductor-generated Triton in this lesson. Starting with Lesson 13, you'll write it. Every concept from Batches 1 and 2 becomes a direct constraint: coalescing rules determine your load patterns, the roofline model tells you if you're memory or compute bound, tiling through shared memory is the pattern for all matrix operations, and the fusion rules tell you how to structure your computation to stay in registers.

Further reading

The TorchInductor design post on the PyTorch dev forum explains the loop-level IR design decisions and why Triton was chosen as the GPU backend over CUDA C. The torch/_inductor/ directory in the PyTorch source is surprisingly navigable — lowering.py maps every aten op to its loop IR, scheduler.py contains the fusion logic, and triton.py handles the code generation. For CUDA Graphs specifically, the NVIDIA CUDA documentation's "CUDA Graphs" chapter explains the capture/replay API and its requirements.
