Why eager mode is slow: the two tax lines

PyTorch's default execution model is eager: every operation launches a CUDA kernel immediately when you call it. y = x @ W launches a matmul kernel right now. z = F.gelu(y) launches a GeLU kernel right now. The result is available before the next line executes. This is wonderful for debugging and interactive development — you get immediate feedback, shapes, values, error messages. It's why PyTorch won the research community.

But eager mode pays two taxes that compound for transformer-scale models.

Tax 1: kernel launch overhead. Every CUDA kernel launch carries approximately 5–15 microseconds of CPU overhead — a CPU-side syscall to submit the kernel to the GPU's work queue. For a transformer forward pass with 100+ operations, that's potentially 1–1.5 ms of pure overhead before a single computation runs. For small operations like bias adds or dropout masks, the kernel launch can take longer than the actual computation. On a T4 doing a 512-token GPT-2 forward pass, kernel launch overhead can be 10–30% of total forward time.

Tax 2: unnecessary VRAM round-trips. Consider three consecutive elementwise operations: y = x + bias, z = F.gelu(y), out = z * gate. Eager execution stores y to VRAM, reads it back for GeLU, stores z to VRAM, reads it back for the multiply. That's 4 VRAM accesses for data that could have stayed in registers the entire time. From Lesson 02: VRAM is 700 cycles. Registers are 1 cycle. Three small ops launch 3 kernels and pay 4 VRAM round-trips when they could be 1 kernel with 0 unnecessary VRAM accesses.

torch.compile eliminates both taxes. It compiles a sequence of operations into a single optimized kernel where adjacent elementwise ops share registers, reducing VRAM round-trips to just the input reads and output writes. And instead of N kernel launches, there's 1. The speedup is 2–4× on transformer forward passes — not from a smarter algorithm, but from removing overhead that was always there.

The three-stage pipeline at 10,000 feet

Stage 01
TorchDynamo — Python interpreter to FX graph
Intercepts Python bytecode execution, traces PyTorch operations into a static directed acyclic graph (FX Graph). Falls back to eager on untraceable code (graph breaks). Output: a pure-functional representation of the computation with no Python control flow.
Stage 02
AOTAutograd — ahead-of-time differentiation
Takes the forward FX graph and traces through PyTorch's autograd machinery symbolically, generating a "joint graph" containing both forward and backward passes. Uses a min-cut algorithm to decide which tensors to save vs recompute during backward. Output: a joint forward+backward FX graph.
Stage 03
TorchInductor — FX graph to fast GPU kernels
Lowers the FX graph to a loop-level IR, schedules and fuses loops, generates Triton code for GPU targets or optimized C++ for CPU. Runs autotuning to find the fastest tile sizes for your specific GPU and input shapes. Output: compiled Triton .py files, cached to disk.

Stage 1 — TorchDynamo: Python to FX graph

Dynamo is the entry point of the compiler. Its job is to take an arbitrary Python function — with all of Python's dynamism, loops, conditionals, and object orientation — and produce a static, pure-functional representation of the operations it performs. This representation is the FX graph: a directed acyclic graph where each node is a PyTorch operation, each edge is a tensor flowing between operations, and there is no Python control flow.

The mechanism: Python exposes a hook called __torch_function__ for intercepting tensor operations, but Dynamo goes deeper. It overrides CPython's frame evaluation function — the C-level function that interprets Python bytecode instruction by instruction. When Dynamo's custom evaluator encounters a PyTorch operation, it records it symbolically into the FX graph instead of executing it. When it encounters Python code it can't trace — a graph break — it falls back to executing that code normally in the Python interpreter, then resumes tracing afterward.

The output is an fx.GraphModule: a Python class whose forward() method contains only the traced operations as explicit function calls, with no loops, no conditionals, no Python side effects. This is the static representation that later stages can analyze and transform.

import torch

def my_fn(x, w, b):
    return torch.nn.functional.gelu(x @ w + b)

# ─── Inspect the FX graph Dynamo produces ─────────────────────────────────────
exported = torch._dynamo.export(my_fn)(
    torch.randn(4, 512),
    torch.randn(512, 512),
    torch.randn(512),
)
exported.graph.print_tabular()
# Output:
# opcode        name         target                  args
# placeholder   x            x                       ()
# placeholder   w            w                       ()
# placeholder   b            b                       ()
# call_function mm           torch.ops.aten.mm.default (x, w)
# call_function add          torch.ops.aten.add.Tensor (mm, b)
# call_function gelu         torch.ops.aten.gelu.default (add,)
# output        output       output                  (gelu,)
#
# The graph is a pure DAG. No Python loops. No conditionals.
# Each node is a single aten op. Edges are implicit from args.

# ─── See what happens with a graph break ──────────────────────────────────────
def fn_with_break(x):
    y = x * 2
    print(y.shape)   # graph break: print() is not traceable
    z = y + 1
    return z

explanation = torch._dynamo.explain(fn_with_break)(torch.randn(4, 512))
print(explanation)
# Shows: 1 graph break at the print() call.
# Two subgraphs: [x*2] and [y+1] compiled separately.
# The print() runs in eager Python between them.

Graph breaks: why they matter

A graph break splits the compilation into multiple smaller subgraphs, each compiled independently. The more breaks, the smaller each compiled region, the fewer operations available to fuse, the less speedup. A model with 20 graph breaks might compile into 21 tiny subgraphs — each graph gets its own kernel launches, and none of the operations across break boundaries can be fused together.

Common causes of graph breaks, in order of frequency:

Python built-ins on tensors: len(tensor), print(tensor), tensor.tolist() — these trigger data-dependent behavior that Dynamo can't trace.

Data-dependent control flow: if tensor.sum() > 0: ... — the branch condition depends on a tensor value only known at runtime, so Dynamo can't trace a single path.

Non-PyTorch operations: numpy calls, custom C extensions, or any code that goes outside the PyTorch ecosystem without a registered tracing rule.

In-place mutation of traced tensors: x[0] = value after x has been traced causes aliasing that Dynamo can't represent in the DAG.

Stage 2 — AOTAutograd: differentiation ahead of time

Standard PyTorch autograd works during the forward pass: as each operation executes, it appends a node to a dynamic "tape" recording what to do during backward. The backward pass then walks this tape in reverse. This runtime graph construction is fundamental to PyTorch's flexibility — it's what allows if statements in your model based on tensor values, and what makes dynamic shapes work naturally.

But runtime graph construction prevents ahead-of-time optimization. You can't fuse across the forward-backward boundary because the backward graph isn't known until the forward pass runs. You can't decide which activations to save vs recompute until you know the backward structure. For compilation, this is a fundamental limitation.

AOTAutograd solves this by tracing through autograd symbolically. It takes the forward FX graph and asks: "if I run this graph with symbolic tensors (placeholders, not real values), what backward operations does autograd register?" The answer is the complete backward graph, represented as FX nodes. Combining forward and backward into a single "joint graph" gives the compiler visibility across both passes simultaneously.

Stage 3 — TorchInductor: FX graph to fast kernels

Inductor is where the actual code generation happens. It takes the joint FX graph from AOTAutograd and compiles it into executable GPU kernels. On GPU targets, it generates Triton code — the same Triton you'll write by hand in Batch 3. Understanding Inductor's output is directly useful for understanding Triton.

Inductor works by lowering the FX graph to a loop-level intermediate representation: a nest of loops over tensor dimensions, where each loop body is a single operation. This IR is simpler to analyze and transform than the FX graph because all tensor operations are expressed as explicit loops with explicit index math. From this loop IR, Inductor applies its key optimization.

Loop fusion: the core transformation

Two adjacent loops that iterate over the same index space can be merged into a single loop. When two operations — say, add and gelu — iterate over the same elements in the same order, their loops are fused: instead of two separate loops (and two separate GPU kernels), there is one loop that computes both operations for each element before moving to the next. The intermediate result never touches VRAM — it flows directly from the add computation into the gelu computation in a register.

This is why Inductor's speedup grows with the number of consecutive elementwise operations. Three fused ops: one kernel launch, two VRAM accesses (input read, output write). Three unfused ops: three kernel launches, six VRAM accesses. For a typical transformer block's post-attention operations — bias add, dropout, residual add, layer norm, linear, GeLU, linear, bias add, dropout, residual add — that's ten operations that Inductor can potentially reduce to one or two kernels.

# ─── Inspect the Triton code Inductor generates ───────────────────────────────
import torch
torch._inductor.config.debug = True   # write generated code to /tmp
torch._inductor.config.trace.enabled = True

def fused_fn(x, bias):
    return torch.nn.functional.gelu(x + bias)

compiled = torch.compile(fused_fn)
compiled(torch.randn(4, 512, device='cuda'),
         torch.randn(512,  device='cuda'))

# Look in /tmp/torchinductor_<user>/ for generated .py files.
# You'll find Triton kernels like:
#
# @triton.jit
# def triton_poi_fused_add_gelu_0(in_ptr0, in_ptr1, out_ptr0, xnumel, XBLOCK):
#     xoffset = tl.program_id(0) * XBLOCK
#     xindex  = xoffset + tl.arange(0, XBLOCK)
#     xmask   = xindex < xnumel
#     x0 = tl.load(in_ptr0 + xindex, xmask)
#     x1 = tl.load(in_ptr1 + (xindex % 512), xmask)  # bias broadcast
#     tmp0 = x0 + x1                                   # fused add
#     # gelu: 0.5 * x * (1 + tanh(sqrt(2/pi) * (x + 0.044715*x^3)))
#     tmp1 = ...gelu math...
#     tl.store(out_ptr0 + xindex, tmp1, xmask)
#
# The add and gelu are a single kernel. One tl.load per input, one tl.store.
# Without compile: two kernels, four VRAM accesses.

Operator fusion: the primary speedup source

Inductor fuses aggressively, but has rules about what it can and can't fuse. Always fuses: adjacent pointwise operations (add, mul, relu, gelu, silu, sigmoid, elementwise comparisons) with matching iteration spaces. Pointwise ops before or after a reduction (sum, mean, max). Does not fuse: operations across different reduction dimensions. Matmuls with subsequent matmuls (different loop structures). Operations separated by a reshape that changes the memory layout.

Understanding fusion rules lets you write models that compile into fewer, larger kernels. The practical implication: if you have multiple consecutive elementwise operations, Inductor will collapse them. If you insert a .contiguous() call in the middle — forcing a memory copy and a layout change — you split the fusion region in two. Write clean functional code with no unnecessary layout manipulations and the compiler does the rest.

Compile modes and when to use each

ModeWhat it doesCompile timeRuntime speedBest for
mode=None (default) Balanced: fuses ops, autotuning for common cases ~30–60 s 2–3× Most training runs. Good all-around tradeoff.
mode="reduce-overhead" Uses CUDA Graphs to eliminate kernel launch overhead for small batches. Captures the entire step as a single GPU command. ~45–90 s 3–4× for small batches Small batch inference. Fixed-shape training loops.
mode="max-autotune" Exhaustive autotuning: benchmarks many tile sizes and kernel configs. Very slow to compile, fastest runtime. 5–30 min 3–5× Production inference. Long training runs where compile cost amortizes.
dynamic=True Traces with symbolic shapes. One compilation handles variable sequence lengths without recompiling per shape. ~60–120 s 1.5–2.5× (slightly less than static) Variable-length batches. Inference with diverse input lengths. 
import torch
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "gpt2", torch_dtype=torch.float16
).cuda()
model.config.use_cache = False  # KV cache breaks graph; disable for training

# ─── Basic compile ────────────────────────────────────────────────────────────
compiled_model = torch.compile(model)  # default mode

# ─── Max-autotune for production inference ────────────────────────────────────
compiled_fast = torch.compile(model, mode="max-autotune")

# ─── Dynamic shapes for variable-length sequences ─────────────────────────────
compiled_dyn  = torch.compile(model, dynamic=True)

# ─── Always warm up before timing ────────────────────────────────────────────
# The first call triggers JIT compilation. Subsequent calls use the cache.
x = torch.randint(0, 50257, (4, 512), device='cuda')
for _ in range(3):   # warmup — first call compiles, second+ uses compiled graph
    compiled_model(x, labels=x).loss.backward()

# ─── Now measure correctly ────────────────────────────────────────────────────
from torch.utils import benchmark

t_eager = benchmark.Timer(
    stmt="model(x, labels=x).loss.backward()",
    globals={"model": model, "x": x}).timeit(50)

t_compiled = benchmark.Timer(
    stmt="compiled_model(x, labels=x).loss.backward()",
    globals={"compiled_model": compiled_model, "x": x}).timeit(50)

print(f"Eager:    {t_eager.mean*1000:.1f} ms")
print(f"Compiled: {t_compiled.mean*1000:.1f} ms")
print(f"Speedup:  {t_eager.mean/t_compiled.mean:.2f}x")
# Expected on T4 with GPT-2: 1.5–2.5× speedup on forward+backward

Your first compiled model: observe the speedup

There is one subtlety that trips up every engineer on their first compile benchmark: the first call always compiles and is always slow. The initial call to a compiled function triggers Dynamo's tracing, AOTAutograd's graph capture, and Inductor's code generation. This takes 30 seconds to several minutes depending on model size and mode. Subsequent calls are fast.

This "cold start" means you must warm up before timing. The minimum warmup is 2 full forward (or forward+backward) passes. The compiled graph is cached both in memory (for the current process) and on disk (in ~/.cache/torch_inductor/). The next time you run the same model with the same input shapes, Inductor reads the cached kernels from disk and compilation is near-instantaneous.

# ─── Full benchmark with correct warmup ──────────────────────────────────────
import torch
from torch.utils import benchmark

def step(model, x):
    with torch.autocast('cuda', torch.float16):
        loss = model(x, labels=x).loss
    loss.backward()
    model.zero_grad(set_to_none=True)
    return loss

# Eager baseline
model.config.use_cache = False
step(model, x)  # warmup for CUDA driver init
t_eager = benchmark.Timer(
    stmt="step(model, x)",
    globals={"step": step, "model": model, "x": x}
).timeit(30)

# Compiled — note the 2-step warmup
cm = torch.compile(model)
step(cm, x)  # warmup 1: triggers compilation (slow!)
step(cm, x)  # warmup 2: uses compiled graph (fast)
t_compiled = benchmark.Timer(
    stmt="step(cm, x)",
    globals={"step": step, "cm": cm, "x": x}
).timeit(30)

print(benchmark.Compare([t_eager, t_compiled]))

# ─── Inspect the generated Triton code ───────────────────────────────────────
import os, glob
cache_dir = os.path.expanduser("~/.cache/torch_inductor")
triton_files = glob.glob(f"{cache_dir}/**/*.py", recursive=True)
print(f"Generated {len(triton_files)} Triton kernel files")
# Read one to see what Inductor generates — it's standard Triton code
if triton_files:
    with open(triton_files[0]) as f:
        print(f.read()[:1000])

Hands-on exercises — Google Colab T4 (free)

Exercise 1: Inspect the FX graph for a simple function. Write a 3-operation function (e.g., linear + layer norm + relu) and use torch._dynamo.export to capture its FX graph. Print the graph with print_tabular(). Count the number of nodes. Verify it matches the number of distinct aten operations you'd expect. Then deliberately introduce a graph break (add a print()) and verify torch._dynamo.explain() reports it.

Exercise 2: Measure the speedup on GPT-2 forward pass. Use torch.utils.benchmark.Timer with proper warmup. Compare eager vs torch.compile(model) for batch sizes 1, 4, 16. At which batch size does compile give the most speedup? (Hypothesis: larger batches have higher arithmetic intensity and benefit more from fusion. Small batches are dominated by kernel launch overhead, so mode="reduce-overhead" helps more there.)

Exercise 3: Find the generated Triton code. Enable torch._inductor.config.debug = True, compile a simple fused function (gelu(x + bias)), and find the generated .py file in the tmp directory. Read the Triton kernel. Identify the tl.load, the fused computation, and the tl.store. Compare to what you'd write by hand — Inductor's output is verbose but functionally identical to a handwritten kernel.

Exercise 4: Count kernel launches, eager vs compiled. Use Nsight Systems or CUDA events to count the number of CUDA kernels launched for one forward pass of your GPT-2 model. Eager should launch O(100) kernels. Compiled with default mode should launch significantly fewer. This is the direct measurement of what fusion is doing.

What comes next

You now understand what torch.compile actually does: Dynamo captures Python into a static FX graph, AOTAutograd traces the backward pass ahead of time and makes smart save/recompute decisions, Inductor fuses adjacent operations into single kernels and generates Triton code. The speedup comes from fewer kernel launches and fewer VRAM round-trips — both consequences of the memory model from Batch 1.

Lesson 08 goes deep into Dynamo — the part of the pipeline that most often fails. Graph breaks are the enemy of compilation, and writing code that compiles cleanly is a skill. Lesson 08 teaches you what causes graph breaks, how to find them with torch._dynamo.explain(), how to fix the most common ones, and what dynamic=True actually does to handle variable-length inputs without endless recompilation.
