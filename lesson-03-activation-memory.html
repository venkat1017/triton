The memory budget you didn't account for

Here is a number that surprises almost every engineer the first time they see it. For Llama-3-8B, the model weights in BF16 occupy 16 GB. But if you try to fine-tune it on a single A100 80G with batch size 1 and sequence length 4096, you run out of memory. The model is 16 GB. The GPU has 80 GB. You should have 64 GB to spare. What's consuming it?

The answer is activations — the intermediate tensors that PyTorch's autograd engine saves during the forward pass so it can compute gradients during the backward pass. For a single training step with batch=1 and seq_len=4096, those saved tensors consume roughly 62 GB for Llama-3-8B. The model weighs 16 GB. The activations weigh 62 GB. The GPU has 80 GB. You're over budget before the optimizer even allocates its state.

This is not a PyTorch bug. It's a fundamental consequence of how reverse-mode automatic differentiation works. To compute the gradient of the loss with respect to a parameter, you need to know the value of every intermediate computation that depended on that parameter. For deep networks, that means the entire forward pass must stay in memory until the backward pass consumes it. Every layer's output. Every attention score. Every pre-activation value.

This lesson builds the exact mental model of what gets saved and why, derives the formula for activation memory from first principles, then explains gradient checkpointing — not as "a thing you turn on" but as a specific engineering trade implemented by a specific mechanism in PyTorch's internals. By the end you'll be able to compute your memory budget before running a single training step, and know exactly what model.gradient_checkpointing_enable() does at the tensor level.

How autograd builds the computation graph

PyTorch's autograd engine implements reverse-mode automatic differentiation using a runtime-constructed directed acyclic graph. During the forward pass, as each operation executes, PyTorch records that operation — its inputs, its output, and which backward function to call when gradients flow through it — into a dynamic computation graph. This graph is the "tape" that reverse-mode AD is often called tape-based differentiation for.

When you call loss.backward(), PyTorch traverses this graph in reverse topological order. At each node, it calls the stored backward function, which computes the gradient of the loss with respect to the node's inputs given the gradient of the loss with respect to the node's output. The key requirement: to compute the backward function for an operation, you often need the original inputs to that operation — inputs that were computed during the forward pass and have since been "consumed" by subsequent operations.

Consider a simple example: y = x * x. The backward function for multiplication is grad_x = grad_y * x + grad_y * x = 2 * grad_y * x. To compute grad_x, you need x — the original input. If x was not stored somewhere, you cannot compute the gradient. Autograd stores it. This is the activation. Every intermediate tensor that a backward function will need is saved into the computation graph node as a "saved tensor."

The deepest problem is the lifetime of these saved tensors. Layer 1's saved activations are needed by layer 1's backward pass. But layer 1's backward pass doesn't run until after layer 32's forward pass (or however deep your network is). So layer 1's activations must remain in VRAM during all 32 forward passes. They aren't freed until the backward pass reaches that layer. For a 32-layer transformer, you simultaneously hold activations from all 32 layers in VRAM at peak memory during the backward pass's first step.

What exactly gets saved: a transformer layer dissection

Let's be precise about which tensors get saved for a single transformer layer. This is where most engineers are fuzzy — they know "activations are expensive" but don't know the specific list. Knowing the list is how you calculate the budget.

A standard transformer layer consists of: self-attention (QKV projection, softmax attention, output projection), a residual connection, layer norm, a feed-forward block (two linear layers with a GeLU in between), and another residual connection. Here's what autograd saves for each piece, assuming BF16 and tensors of shape [batch, seq_len, d_model]:

Operation What gets saved Shape Why it's needed in backward
LayerNorm (pre-attn) input x, mean, rstd [B, S, D] + [B, S] + [B, S] grad requires normalized input and the normalization statistics
QKV projection input x (post-norm) [B, S, D] grad_W = x.T @ grad_output
Attention scores (Q·KT) Q, K, softmax output [B, H, S, D/H] × 2 + [B, H, S, S] backward through softmax and matmul needs both operands
Attention output (·V) V, attention weights [B, H, S, D/H] + [B, H, S, S] grad_V = attn_weights.T @ grad; grad_attn = grad @ V.T
Output projection attention output [B, S, D] grad_W = attn_out.T @ grad
LayerNorm (pre-FFN) input x, mean, rstd [B, S, D] + [B, S] + [B, S] same as pre-attention LayerNorm
FFN first linear input x (post-norm) [B, S, D] grad_W1 = x.T @ grad
GeLU pre-activation (W1 output) [B, S, 4D] grad requires the pre-activation value to compute gelu'(x)
FFN second linear GeLU output [B, S, 4D] grad_W2 = gelu_out.T @ grad
Dropout (if used) dropout mask [B, S, D] bool backward must apply same mask

The two most expensive entries in that table are the attention score matrices [B, H, S, S] (the full N×N attention matrix), which appear twice, and the FFN intermediate values at 4× the model dimension. The attention score matrix scales quadratically with sequence length. At S=2048, H=32, B=1, in BF16: 1 × 32 × 2048 × 2048 × 2 bytes = 268 MB just for the softmax output. At S=8192 that's 4.3 GB. Per layer. This is the dominant reason standard attention is memory-explosive at long contexts — and precisely the problem Flash Attention (Lesson 18) eliminates by recomputing attention scores during the backward pass instead of saving them.

The N² activation is the principal target

When you hear that Flash Attention is memory-efficient, this is what it means specifically. Standard attention saves the full [B, H, S, S] softmax matrix for the backward pass. Flash Attention recomputes it during backward from Q, K (which are already needed for other reasons) — paying compute instead of memory. For S=8192 and 32 layers: standard attention's activation cost for just this one tensor is 32 × 4.3 GB = 137 GB. Flash Attention's activation cost for this tensor is 0 GB. The rest of the memory picture is the same.

Non-attention activations per layer: Roughly B × S × D × 10 × 2 bytes (accounting for QKV input, V tensor, output projection input, both LayerNorm inputs, FFN input, pre-GeLU, post-GeLU). That's 1 × 4096 × 4096 × 10 × 2 = 335 MB per layer, or 32 × 335 MB ≈ 10.7 GB total.

Attention score matrices per layer: The full Q·KT before softmax and the softmax output (both needed for backward). Shape: [B, H, S, S] × 2 = 1 × 32 × 4096 × 4096 × 2 × 2 bytes = 4.3 GB per layer, or 32 × 4.3 GB = 137 GB total. The A100 has 80 GB. This is why standard attention cannot train at S=4096 on Llama-3-8B on any single GPU — not even close.

With Flash Attention (which is what HuggingFace uses by default when you install flash-attn), the N² term drops to zero. Activation memory becomes dominated by the non-attention terms: approximately 10.7 GB. Suddenly the A100 can do it.

The complete training memory budget for Llama-3-8B

Knowing activation memory in isolation is not enough. Training memory has five components and they all run simultaneously. You need to account for all of them to know whether a configuration fits on your GPU.

The numbers reveal a counterintuitive fact: for large models, the optimizer state dominates the memory budget, not the activations. Adam's first and second moment buffers are maintained in FP32 (even when training in BF16) because the optimizer step requires precision that BF16 cannot provide. Two FP32 tensors at 4 bytes each, for 8 billion parameters: 2 × 8 × 10⁹ × 4 = 64 GB. That's 4× the model weights.

This is why LoRA and QLoRA are so effective: they don't just reduce the compute cost of training. They reduce the optimizer state cost proportionally to the number of trainable parameters. LoRA adds ~1% of the model's parameters as trainable adapters. The optimizer state for those adapters is 1% of 64 GB = 640 MB. The rest of the budget can be used for batch size or sequence length.

Gradient checkpointing: the mechanism

Gradient checkpointing is a technique that trades compute for memory. Instead of saving all intermediate activations during the forward pass, you save only certain "checkpoint" tensors at designated boundaries, then recompute the intermediate activations during the backward pass from those checkpoints as needed.

Here is the exact mechanism PyTorch uses, step by step:

During the forward pass: When execution enters a checkpointed segment (a function wrapped with torch.utils.checkpoint.checkpoint()), PyTorch runs that function normally but with one modification: it wraps the execution in a context that suppresses autograd's normal graph-building. The intermediate activations are computed and used, but the computation graph nodes that would normally save them are not created. Only the input tensors to the segment and the output tensors are kept. Everything in between is immediately freed.

What's actually saved: The inputs to each checkpoint segment. These are the "checkpoints" — the tensors from which the segment can be re-executed. For a transformer with 32 layers where each layer is one checkpoint, you save 32 × 1 × input tensors instead of 32 × ~10 intermediate tensors per layer.

During the backward pass: When the backward pass reaches a checkpoint boundary and needs the intermediate activations from that segment to compute gradients, it re-runs the forward function. This time, with autograd enabled, so the computation graph is built normally and the intermediate activations are created fresh. The backward through this segment then proceeds using those freshly computed activations. When it's done, those activations are freed again.

The memory-compute tradeoff, quantified

The memory reduction from gradient checkpointing is large. The compute cost is fixed and predictable. Let's quantify both.

Memory savings: For a transformer layer that saves approximately 10 intermediate tensors (each of shape [B, S, D]), checkpointing replaces those 10 tensors with 1 checkpoint tensor (the layer input, also [B, S, D]). That's a 10× reduction in activation memory per layer. For Llama-3-8B with Flash Attention and no checkpointing: 10.7 GB activations. With checkpointing: approximately 10.7 / 10 ≈ 1.1 GB. In practice, the savings are 3–5× rather than 10× because some activations (like the LayerNorm statistics and the residual connection values) are still saved by the outer computation graph even when a layer is checkpointed.

Compute cost: During the backward pass, each checkpointed segment runs its forward pass once more. For a network with N checkpointed segments, that's N additional forward passes during backward. Since the backward pass already costs roughly 2× the forward pass, the total training step cost goes from 1 (forward) + 2 (backward) = 3 units to approximately 1 (forward) + 1 (recompute) + 2 (backward) = 4 units. A 33% increase in total compute per training step. For most training runs, this is the correct tradeoff: memory is the binding constraint, and compute is available.

The compute cost is not proportional to how many layers you checkpoint. Checkpointing all 32 layers of Llama-3-8B still only costs one extra forward pass — because the backward pass already traverses all 32 layers, and the recomputation is interleaved with the backward traversal, not added on top. The recomputed activations from layer 1 are computed, consumed by the backward function, and freed before layer 2's recomputation begins.

Implementation: torch.utils.checkpoint in practice

There are two ways to use gradient checkpointing in PyTorch. The first is the low-level function API. The second is the high-level model API. Both do the same thing; the model API is a convenience wrapper.

import torch
from torch.utils.checkpoint import checkpoint

# ─── Low-level API: wrap any callable ────────────────────────────────────────
def transformer_layer_forward(x, layer):
    # Without checkpointing: every intermediate in here gets saved by autograd.
    # With checkpointing: only x (the input) is saved. Everything else is freed
    # and recomputed during backward.
    return checkpoint(
        layer,           # the callable to checkpoint
        x,               # its argument (the saved checkpoint tensor)
        use_reentrant=False  # always use False for new code (see note below)
    )

# ─── High-level API: enable on any HuggingFace model ─────────────────────────
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# This wraps every transformer block's forward with checkpoint().
# After this call, training activation memory drops ~3-5× per layer.
model.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={"use_reentrant": False}
)

# Critical: disable KV cache for training (checkpointing + KV cache conflict)
model.config.use_cache = False

# ─── What gradient_checkpointing_enable() does internally ────────────────────
# It modifies the forward method of each transformer block to call checkpoint().
# Roughly equivalent to:
#
#   original_forward = layer.forward
#   def checkpointed_forward(*args, **kwargs):
#       return checkpoint(original_forward, *args,
#                         use_reentrant=False, **kwargs)
#   layer.forward = checkpointed_forward
use_reentrant=False — always use this

The original gradient checkpointing implementation (use_reentrant=True, the default) has a subtle bug: it doesn't correctly handle all cases of nested checkpointing, and it's incompatible with torch.compile because it relies on a re-entrant backward hook mechanism that the compiler can't trace through. use_reentrant=False uses a cleaner implementation that works correctly with nested checkpoints and with torch.compile. There is no reason to use True in new code. HuggingFace's gradient_checkpointing_enable() accepts this as a keyword argument in versions ≥ 4.38.

Selective checkpointing

You don't have to checkpoint every layer. For a 32-layer model, you might checkpoint every other layer — halving activation memory at the cost of 0.5× forward recompute instead of 1×. The optimal granularity depends on your memory budget and compute budget. The math is straightforward:

# Checkpoint every k-th layer. k=1: checkpoint all (max savings, max cost).
# k=2: checkpoint every other layer (half savings, half cost). Etc.

for i, layer in enumerate(model.transformer.layers):
    if i % 2 == 0:  # checkpoint even-numbered layers
        original_fwd = layer.forward
        layer.forward = lambda *args, layer=layer, **kw: checkpoint(
            original_fwd, *args, use_reentrant=False, **kw
        )

# More explicit: wrapping a specific segment within a layer.
# This is useful when only the attention or only the FFN is the memory hog.

class CheckpointedAttention(torch.nn.Module):
    def __init__(self, attn_module):
        super().__init__()
        self.attn = attn_module

    def forward(self, x, *args, **kwargs):
        # Only checkpoint the attention computation, not the FFN
        return checkpoint(self.attn, x, *args,
                          use_reentrant=False, **kwargs)

Measuring activation memory precisely

Everything above is theory. The most important engineering habit is measuring it. PyTorch provides the exact tools to see activation memory as a number, not an estimate. Here is the canonical measurement procedure:

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained(
    "gpt2",  # use GPT-2 for free T4; swap for larger model on A100
    torch_dtype=torch.float16
).cuda()
model.config.use_cache = False

# ─── Measure activation memory for one forward pass ──────────────────────────
def measure_activation_memory(model, input_ids, use_checkpoint=False):
    if use_checkpoint:
        model.gradient_checkpointing_enable(
            gradient_checkpointing_kwargs={"use_reentrant": False}
        )
    else:
        model.gradient_checkpointing_disable()

    torch.cuda.synchronize()
    torch.cuda.reset_peak_memory_stats()

    before = torch.cuda.memory_allocated()

    # Forward pass with gradient tracking
    output = model(input_ids, labels=input_ids)
    loss   = output.loss

    torch.cuda.synchronize()
    after_fwd = torch.cuda.memory_allocated()

    # The difference is activation memory (tensors saved for backward)
    activation_mem = (after_fwd - before) / 1024**3

    # Now run backward and measure peak
    loss.backward()
    torch.cuda.synchronize()
    peak = torch.cuda.max_memory_allocated() / 1024**3

    return activation_mem, peak

# Prepare input
tokenizer = AutoTokenizer.from_pretrained("gpt2")
text      = "The GPU memory hierarchy " * 128  # ~512 tokens
input_ids = tokenizer.encode(text, return_tensors="pt").cuda()

# Measure without checkpointing
act_no_ckpt, peak_no_ckpt = measure_activation_memory(model, input_ids, False)
print(f"No checkpointing: activations={act_no_ckpt:.2f} GB, peak={peak_no_ckpt:.2f} GB")

# Measure with checkpointing
model.zero_grad(set_to_none=True)
act_ckpt, peak_ckpt = measure_activation_memory(model, input_ids, True)
print(f"Checkpointing:    activations={act_ckpt:.2f} GB, peak={peak_ckpt:.2f} GB")
print(f"Activation reduction: {act_no_ckpt/act_ckpt:.1f}x")

The memory snapshot: seeing every allocation

The numbers above tell you the peak. The memory snapshot tells you which operations allocated that memory and exactly when. This is the tool you reach for when the peak memory is higher than you expect and you need to diagnose what's holding it up.

import pickle

# Enable recording (captures stack traces for every allocation)
torch.cuda.memory._record_memory_history(max_entries=100_000)

# Run the training step you want to profile
output = model(input_ids, labels=input_ids)
output.loss.backward()

# Capture the snapshot
snapshot = torch.cuda.memory._snapshot()
print("Saving memory snapshot...")
with open("memory_snapshot.pickle", "wb") as f:
    pickle.dump(snapshot, f)

# Stop recording
torch.cuda.memory._record_memory_history(enabled=None)

# Now upload memory_snapshot.pickle to pytorch.org/memory_viz
# The visualization shows:
#   Y axis: VRAM used (GB)
#   X axis: allocation events (time)
#   Color: allocation type (activations, parameters, gradients, etc.)
#   Hover: Python stack trace that triggered each allocation
#
# What to look for:
#   1. The peak — this is your OOM threshold
#   2. The large plateau during forward pass — this is activation memory
#   3. The gradual release during backward — activations freed layer by layer
#   4. Any unexpected allocations outside the model forward (logging? callbacks?)

# On Colab, download with:
from google.colab import files
files.download("memory_snapshot.pickle")
Reading the memory_viz output

The most important pattern to identify in the visualization is the "activation plateau": a large step up in memory at the start of the forward pass that stays constant throughout all the forward layers, then slowly decreases during the backward pass as each layer's activations are consumed. The height of that plateau is your activation memory. If the plateau persists well into the backward pass without decreasing, it usually means a layer's saved tensors are not being released — often because a hook or external reference is keeping them alive.

Three common memory leaks in training loops

While the snapshot tool is for deep diagnosis, there are three mistakes that cause memory growth per step — detectable with a simple memory_allocated() call in your training loop:

# ─── Leak 1: storing loss tensors instead of scalars ─────────────────────────
losses = []
for step in range(100):
    loss = model(input_ids, labels=input_ids).loss
    losses.append(loss)          # BAD: keeps entire compute graph alive
    losses.append(loss.item())  # GOOD: extracts scalar, releases graph

# ─── Leak 2: not freeing gradients properly ───────────────────────────────────
optimizer.zero_grad()              # BAD: zeros gradient tensors but keeps them
optimizer.zero_grad(set_to_none=True)  # GOOD: frees gradient tensors entirely

# ─── Leak 3: hooks that capture activations ───────────────────────────────────
outputs = []
def hook(module, input, output):
    outputs.append(output)           # BAD: output holds the compute graph
    outputs.append(output.detach().cpu())  # GOOD: detach cuts the graph

model.some_layer.register_forward_hook(hook)

# ─── Diagnosis: check if memory grows per step ───────────────────────────────
for step in range(5):
    loss = model(input_ids, labels=input_ids).loss
    loss.backward()
    optimizer.step()
    optimizer.zero_grad(set_to_none=True)
    mem = torch.cuda.memory_allocated() / 1024**2
    print(f"Step {step}: {mem:.1f} MB")
    # If this number grows step-over-step, you have a memory leak.
    # If it stabilizes after step 1 or 2, you're clean.

What comes next

You now have a precise mental model of training memory. Weights and gradients are one copy of the parameters each — fixed cost. Optimizer state is two copies in FP32 — the dominant cost for large models. Activations are the thing you control through Flash Attention and gradient checkpointing. The N² attention score term is the activation's dominant driver at long sequences. Gradient checkpointing trades 33% compute to eliminate ~8–10× of activation memory. You can calculate all of these numbers before writing a single line of training code.

Lesson 04 goes into the precision question: what exactly are FP16 and BF16 at the bit level, why does FP16 overflow during training while BF16 doesn't, what does torch.autocast() actually cast and what does it leave in FP32, and where does FP8 fit into this picture. Understanding precision is what turns "add fp16=True" from a magic flag into a tool you know how to use correctly — and debug when it breaks.
