AOTAutograd: Ahead-of-Time Differentiation

Understand how AOTAutograd traces both forward and backward passes into a single joint graph — and why that enables cross-pass optimization that standard autograd cannot.

The problem with runtime autograd

Standard PyTorch autograd is a tape-based system. During the forward pass, every differentiable operation appends a record to a dynamically growing computation graph: what the operation was, what its inputs were, which backward function to call when gradients flow through. This tape is constructed one operation at a time, at runtime, with real tensor values.

This design gives PyTorch its flexibility. You can write if loss.item() > 10: scale_factor = 0.1 inside a training loop, and autograd handles it naturally — the tape just records whatever path execution actually took. Dynamic computation graphs are one of the core reasons PyTorch became the research standard: you can build networks that change structure based on input, iterate over variable-length sequences, and use arbitrary Python control flow.

But for compilation, runtime graph construction is the enemy. The problem has two layers.

Layer 1: the backward pass is unknowable before the forward runs. The compiler cannot see the backward graph until after the forward pass executes, because the backward graph is constructed during the forward pass from the actual operations performed. There is no static structure to analyze. This prevents any optimization that spans the forward-backward boundary — the compiler is essentially blind to half the computation.

Layer 2: saved tensors are chosen naively. Standard autograd saves tensors based on local rules: each operation's backward function declares what it needs, and autograd saves exactly that. There is no global view — no optimizer that looks at the whole network and asks "given that I'm going to need these 50 tensors for backward, which ones are cheapest to recompute vs save?" Every tensor is saved unless you explicitly use gradient checkpointing, and checkpointing is a coarse instrument (whole layers, not individual operations).

AOTAutograd solves both problems by tracing the backward pass symbolically, before the first actual execution, using PyTorch's functional programming infrastructure.

How AOT tracing works: functorch under the hood

AOTAutograd builds on functorch — PyTorch's functional transforms library. The key primitive is torch._functorch.aot_autograd, which takes a forward function and produces a pair of functions: a compiled forward and a compiled backward, both represented as FX graphs.

The mechanism: instead of running the forward function with real tensors, AOTAutograd runs it with proxy tensors — abstract objects that record every operation they participate in but contain no actual data. When the forward function returns a loss (also a proxy tensor), AOTAutograd calls loss.backward() on this proxy. PyTorch's autograd machinery executes normally, walking the tape it built during the proxy forward pass, calling each operation's backward function with proxy gradient tensors. The result: the entire backward computation is traced into FX nodes, exactly as the forward was.

The output is a single joint graph containing both the forward and backward computations as a unified DAG, with no runtime tape, no dynamic dispatch, no Python overhead.

The joint graph: forward + backward as one structure

The joint graph is a single FX graph containing all operations from both the forward and backward passes. It has a clear structure: all forward nodes come first, then a "partition boundary" (conceptually, not literally in the graph), then all backward nodes. Data flows forward through the graph for the forward computation, and backward nodes consume both their incoming gradient tensors and any saved tensors from the forward section.

What makes this powerful is what the compiler can now see across the boundary. Consider a GeLU activation: in the forward pass, it computes gelu(x). In the backward pass, it needs x (the pre-activation value) to compute the derivative. Standard autograd saves x unconditionally, paying 1 VRAM write during forward and 1 VRAM read during backward. But the joint graph lets Inductor see that x was already computed in the forward section — and more importantly, it was computed cheaply (it's just the output of a preceding linear layer). Perhaps it's cheaper to recompute it from the linear layer's inputs than to store it. The joint graph makes this analysis possible.

from torch._functorch.aot_autograd import aot_export_module
import torch
import torch.nn as nn

# ─── Export the joint graph for a small MLP ──────────────────────────────────
class TwoLayerMLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(256, 512)
        self.fc2 = nn.Linear(512, 256)

    def forward(self, x):
        return self.fc2(nn.functional.gelu(self.fc1(x)))

model = TwoLayerMLP()
x     = torch.randn(4, 256)

# Export the joint graph (forward + backward as a single FX GraphModule)
joint_module, guards = aot_export_module(
    model,
    (x,),
    trace_joint=True   # include backward in the graph
)
joint_module.graph.print_tabular()

# The output will show all forward ops (mm, add, gelu, mm, add)
# followed by backward ops (gelu_backward, mm_backward, etc.)
# and importantly — which tensors cross the partition boundary:
# those are the "saved tensors" that standard autograd would save.

# Count saved tensors (nodes that flow from forward section to backward section)
fwd_nodes = [n for n in joint_module.graph.nodes if 'primals' in n.name]
bwd_nodes = [n for n in joint_module.graph.nodes if 'tangents' in n.name or 'grad' in n.name]
print(f"Forward nodes: {len(fwd_nodes)}")
print(f"Backward nodes: {len(bwd_nodes)}")

Min-cut partitioning: save vs recompute

The most technically sophisticated part of AOTAutograd is the min-cut partitioning algorithm. It answers the question: for each intermediate tensor computed in the forward pass that the backward pass needs, is it cheaper to save it (write once to VRAM during forward, read once during backward) or recompute it (don't save it; re-run the forward computation that produced it during the backward pass when it's needed)?

This is a graph optimization problem. The joint graph has a set of "boundary tensors" — values that must cross from the forward section to the backward section. Reducing the number and size of boundary tensors reduces activation memory. But recomputing a tensor during backward costs compute time. The min-cut algorithm finds the optimal boundary: the cut through the graph that minimizes the total cost of boundary tensors, where cost accounts for both tensor size (memory) and recomputation expense (compute).

How the min-cut cost model works

The min-cut algorithm assigns a cost to each potential boundary tensor. The cost has two components: the memory cost (tensor size in bytes — larger tensors are more expensive to save) and the recomputation cost (FLOPs required to reproduce the tensor from an earlier saved value — operations with high arithmetic intensity are expensive to recompute).

Heuristics used by the default Inductor min-cut implementation:

Always save: tensors that are expensive to recompute (matmul outputs, attention scores with softmax applied) and whose size is manageable.

Recompute if cheap: tensors produced by elementwise operations (adds, multiplies, activations like GeLU, layer normalization) that can be regenerated quickly from a saved ancestor. Recomputing GeLU from its input costs roughly the same as a memory read of the saved output — so recomputation may be preferred.

Always save: the model's input tensors and weight tensors — these are always needed by backward functions and can't be recomputed (they don't have "ancestors" in the graph).

Viewing the joint graph yourself

Understanding AOTAutograd becomes concrete when you look at the actual joint graph for a model you care about. Here is the full workflow for examining it on GPT-2:

import torch
from transformers import AutoModelForCausalLM
from torch._functorch.aot_autograd import aot_export_module

model = AutoModelForCausalLM.from_pretrained("gpt2")
model.config.use_cache = False
x = torch.randint(0, 50257, (1, 32))

# ─── Export the joint graph ───────────────────────────────────────────────────
joint_gm, guards = aot_export_module(model, (x,), trace_joint=True)

# Count forward vs backward ops
all_nodes = list(joint_gm.graph.nodes)
fwd_ops   = [n for n in all_nodes if n.op == 'call_function' and 'tangent' not in ''.join([str(a) for a in n.args])]
print(f"Total graph nodes:  {len(all_nodes)}")

# ─── Identify the saved tensors (cross-partition boundary) ────────────────────
# AOTAutograd marks partition boundary; we can see it in the graph structure.
# A simpler proxy: count 'primals' (forward tensors) and 'tangents' (grad tensors)
primals  = [n for n in all_nodes if 'primals'  in n.name]
tangents = [n for n in all_nodes if 'tangents' in n.name]
print(f"Primal inputs:      {len(primals)}")    # weights + input
print(f"Tangent inputs:     {len(tangents)}")   # grad_output for backward

# ─── Print the first 20 nodes to see the structure ───────────────────────────
for i, node in enumerate(all_nodes[:20]):
    print(f"{i:3d} {node.op:15} {node.name:30} target={str(node.target)[:40]}")
What to look for in the joint graph

After printing the joint graph for GPT-2, look for two things: (1) the density of nodes — GPT-2 has 12 transformer blocks, and each block's operations appear as individual nodes, giving a graph with hundreds of nodes for a model that eager mode would handle with a handful of Python function calls; (2) the saved tensor boundary — nodes that are placeholders for the partition crossing are the tensors AOTAutograd decided to materialize. Compare the count to what naïve save-everything would have saved from your mental model in Lesson 03.

AOTAutograd and functional transforms

AOTAutograd is built on the same functional transform infrastructure as torch.vmap, torch.func.grad, and torch.func.jacrev. These transforms are all implemented as function compositions over the same underlying tracing machinery. Understanding this explains why they compose cleanly and why you can write code like torch.func.vmap(torch.func.grad(loss_fn)) without special cases.

The key abstraction is the functional interpreter: a system that takes a function f(x) and a transform T, and produces a new function T(f)(x) that is itself traceable. AOTAutograd is the transform T = aot. Vmap is the transform T = vectorize. Grad is the transform T = differentiate. Because they all speak the same proxy tensor language, you can compose them: aot(vmap(grad(f))) is a valid, compilable expression.

import torch
from torch import func

# ─── Functional transforms compose with torch.compile ─────────────────────────
def loss_fn(params, x, y):
    # params is a dict of parameter tensors (functional style)
    pred = torch.nn.functional.linear(x, params['weight'], params['bias'])
    return torch.nn.functional.mse_loss(pred, y)

# grad computes the gradient w.r.t. params (first argument)
grad_fn = func.grad(loss_fn)

# vmap vectorizes over a batch dimension — compute per-sample gradients efficiently
per_sample_grads = func.vmap(grad_fn, in_dims=(None, 0, 0))

# torch.compile works across all of this because it's all proxy-tensor-based
compiled_per_sample_grads = torch.compile(per_sample_grads)

params = {'weight': torch.randn(10, 20), 'bias': torch.randn(10)}
x_batch = torch.randn(32, 20)  # batch of 32 samples
y_batch = torch.randn(32, 10)

# Per-sample gradients in one compiled forward+backward pass
grads = compiled_per_sample_grads(params, x_batch, y_batch)
print({k: v.shape for k, v in grads.items()})
# {'weight': torch.Size([32, 10, 20]), 'bias': torch.Size([32, 10])}
# 32 separate gradients computed in one vectorized compiled pass.
# This is the pattern used in differential privacy (DP-SGD) and NTK computations.

Diagnosing compile regressions with AOT

The most common use of directly examining AOTAutograd's output is diagnosing "compile regressions" — cases where torch.compile is slower than expected or produces incorrect results. The joint graph is the bridge between "Dynamo captured the wrong thing" and "Inductor generated slow code."

# ─── Diagnosing a regression: what did AOT actually see? ─────────────────────

# Step 1: Confirm Dynamo is giving AOT the right graph
explanation = torch._dynamo.explain(model)(x)
print(f"Break count: {len(explanation.break_reasons)}")
# If there are breaks, fix them first — AOT can't help with fragmented graphs

# Step 2: Export the joint graph and sanity-check its size
joint_gm, _ = aot_export_module(model, (x,), trace_joint=True)
node_count   = sum(1 for n in joint_gm.graph.nodes if n.op == 'call_function')
print(f"Total call_function nodes: {node_count}")
# For GPT-2 (12 layers): expect ~400-600 nodes for fwd+bwd
# For Llama-7B (32 layers): expect ~2000-4000 nodes
# If the number is much higher: Dynamo may have unrolled loops it shouldn't
# If the number is much lower: graph breaks truncated the graph

# Step 3: Look for unexpected saved tensors
# After partition, list the nodes that cross the boundary
# These are stored in the graph metadata
if hasattr(joint_gm, '_saved_tensors_hooks'):
    print("Saved tensors:", joint_gm._saved_tensors_hooks)

# Step 4: Enable AOT verbose logging to see partition decisions
import logging
logging.getLogger("torch._functorch.aot_autograd").setLevel(logging.DEBUG)
joint_gm2, _ = aot_export_module(model, (x,), trace_joint=True)
# This logs each partitioning decision:
# "Decided to save tensor X (size: Y MB, recompute cost: Z GFLOPs)"
# "Decided to recompute op A (cost: B GFLOPs, saves: C MB)"

# Standard autograd: inspect what's saved on the tape
x = torch.randn(4, 256, requires_grad=True)
y = model(x)
loss = y.sum()

# Walk the grad_fn chain to count saved tensors
def count_saved(grad_fn, visited=None):
    if visited is None: visited = set()
    if grad_fn is None or id(grad_fn) in visited: return 0
    visited.add(id(grad_fn))
    saved = len(grad_fn.saved_tensors()) if hasattr(grad_fn, 'saved_tensors') else 0
    return saved + sum(count_saved(nf[0], visited) for nf in grad_fn.next_functions)

print(f"Standard autograd saved tensors: {count_saved(loss.grad_fn)}")

What comes next

You now understand what AOTAutograd produces and why. The joint graph gives Inductor visibility over both passes. The min-cut partitioning makes intelligent save/recompute decisions that reduce activation memory compared to standard autograd. The functional transform composition explains why vmap and grad work with compile.
