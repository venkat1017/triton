Why time.time() is wrong for GPU code

GPU operations are asynchronous. When you call y = model(x), the CUDA runtime queues the computation on the GPU's work stream and immediately returns control to the CPU. The GPU begins executing in parallel. The Python line after y = model(x) runs on the CPU while the GPU is still computing y.

This means:

import time

start = time.time()
y = model(x)         # queues work, returns immediately
end = time.time()    # measures time to QUEUE, not time to COMPUTE

elapsed = end - start  # THIS IS WRONG for measuring GPU performance
# For a large model: elapsed ≈ 0.0002s (queue overhead)
# Actual computation: 0.050s (still running when you measured!)

The elapsed you measure is the time for the CPU to submit the kernel launch commands to the GPU's work queue — typically 0.1–5 ms regardless of how long the GPU computation actually takes. For a large model that takes 100 ms to run, you'd measure 2 ms and conclude the model is 50× faster than it is. This mistake is so common it's worth calling out explicitly: every GPU benchmark you've seen in a blog post that uses time.time() is measuring the wrong thing.

CUDA synchronization: the missing primitive

torch.cuda.synchronize() blocks the CPU until all currently queued GPU kernels have completed. It's the essential primitive for correct GPU timing. Call it before starting a timer and after the work you're measuring:

import time, torch

# ─── Correct time.time() usage with explicit sync ────────────────────────────
torch.cuda.synchronize()       # wait for any pending work from before
start = time.time()
y = model(x)
torch.cuda.synchronize()       # wait for y to actually be computed
end = time.time()
elapsed = end - start            # NOW this measures actual computation time

# This works but has two problems:
# 1. synchronize() itself has overhead (~0.1-0.5ms) that pollutes the measurement
# 2. Running one iteration gives noisy results — you need statistical aggregation
# The right tool handles both: torch.utils.benchmark.Timer
synchronize() stalls the pipeline

Calling torch.cuda.synchronize() inside a production training loop is a performance anti-pattern. It forces the CPU to wait for the GPU, serializing what should be an asynchronous pipeline. Use it only for benchmarking or when you explicitly need a result on the CPU (e.g., for a conditional branch based on the loss value). The benchmark tools handle synchronization internally, so you don't need to add it yourself when using benchmark.Timer.

torch.utils.benchmark.Timer: the correct tool

torch.utils.benchmark.Timer handles everything correctly: it runs a configurable warmup, synchronizes before and after each measurement, runs multiple iterations for statistical stability, and reports the mean, median, and interquartile range. It's the tool to use for all GPU performance comparisons in this course.

from torch.utils import benchmark
import torch

model = ...  # your compiled or eager model
x     = torch.randint(0, 50257, (4, 512), device='cuda')

# ─── Basic Timer usage ────────────────────────────────────────────────────────
t = benchmark.Timer(
    stmt="model(x)",             # string to execute
    globals={"model": model, "x": x},
    num_threads=1,                # CPU threads (keep at 1 for GPU work)
    label="GPT-2 forward",        # shown in comparison tables
    sub_label="eager, batch=4",   # shown as a row label
)

result = t.timeit(100)   # 100 iterations; Timer handles warmup internally

print(f"Mean:   {result.mean*1000:.2f} ms")
print(f"Median: {result.median*1000:.2f} ms")
print(f"IQR:    {result.iqr*1000:.3f} ms  (lower = more stable)")

# ─── Blocked runs for higher statistical precision ───────────────────────────
result_blocked = t.blocked_autorange(min_run_time=5.0)  # runs until 5 sec elapsed
# Automatically determines the number of iterations needed for stable results.
# Use this when you need high-confidence measurements.

What Timer does internally

Understanding Timer's internals prevents misuse. It wraps your stmt in a loop, and for each iteration it:

1. Calls torch.cuda.synchronize() before the measurement starts

2. Executes the stmt string via Python's exec()

3. Calls torch.cuda.synchronize() after to wait for GPU completion

4. Records the wall-clock time for that iteration

5. Applies statistical aggregation over all iterations

The implicit warmup: timeit(N) does not explicitly warm up. The first few iterations include JIT compilation (for compiled models) and CUDA cold-start overhead. For compiled models, always call the model a few times before constructing and running the Timer. This is why Lesson 11 showed the 3-step warmup as a prerequisite to benchmarking.

CUDA Events: microsecond-precision kernel timing

For measuring individual kernel performance — not a full forward pass, but a specific operation — CUDA Events provide higher precision than benchmark.Timer because they record timestamps on the GPU itself, avoiding any CPU clock granularity or synchronization overhead.

# ─── CUDA Events: measure exactly the GPU execution time ─────────────────────
start_event = torch.cuda.Event(enable_timing=True)
end_event   = torch.cuda.Event(enable_timing=True)

# Warmup: run the kernel a few times first
for _ in range(3):
    y = model(x)

# Record timestamps around the operation of interest
start_event.record()        # GPU timestamp: "when did this point in the stream occur?"
y = model(x)
end_event.record()          # GPU timestamp after the operation

torch.cuda.synchronize()   # must sync before reading elapsed_time
elapsed_ms = start_event.elapsed_time(end_event)
print(f"GPU execution time: {elapsed_ms:.3f} ms")

# ─── For averaging over many runs ────────────────────────────────────────────
N_RUNS = 100
events = [(torch.cuda.Event(enable_timing=True),
           torch.cuda.Event(enable_timing=True)) for _ in range(N_RUNS)]

for start, end in events:
    start.record()
    model(x)
    end.record()

torch.cuda.synchronize()
times_ms = [s.elapsed_time(e) for s, e in events]

import statistics
print(f"Mean:   {statistics.mean(times_ms):.3f} ms")
print(f"Median: {statistics.median(times_ms):.3f} ms")
print(f"Stdev:  {statistics.stdev(times_ms):.3f} ms")
print(f"Min:    {min(times_ms):.3f} ms  (← theoretical best case)")
# The minimum is often the most informative: it represents the fastest possible
# execution without any OS scheduling or thermal throttling interference.

Comparing implementations: benchmark.Compare

benchmark.Compare takes a list of Measurement objects and renders a formatted table with automatic speedup ratios. This is the right format for presenting results — not "model A took 10ms and model B took 7ms," but a table that shows the comparison at every batch size and sequence length combination simultaneously.

from torch.utils import benchmark

results = []

for batch_size in [1, 4, 16, 32]:
    for seq_len in [128, 512, 1024]:
        x = torch.randint(0, 50257, (batch_size, seq_len), device='cuda')

        # Eager baseline
        results.append(benchmark.Timer(
            stmt="model(x)",
            globals={"model": eager_model, "x": x},
            label="GPT-2 forward",
            sub_label=f"B={batch_size}, S={seq_len}",
            description="eager",
        ).timeit(50))

        # Compiled with default mode
        results.append(benchmark.Timer(
            stmt="model(x)",
            globals={"model": compiled_model, "x": x},
            label="GPT-2 forward",
            sub_label=f"B={batch_size}, S={seq_len}",
            description="compiled",
        ).timeit(50))

        # Compiled with reduce-overhead (CUDA Graphs)
        results.append(benchmark.Timer(
            stmt="model(x)",
            globals={"model": compiled_cg, "x": x},
            label="GPT-2 forward",
            sub_label=f"B={batch_size}, S={seq_len}",
            description="compiled+cudagraphs",
        ).timeit(50))

compare = benchmark.Compare(results)
compare.trim_significant_figures()
compare.colorize(rowwise=True)  # colors fastest in each row green
compare.print()
# Output:
# [------------- GPT-2 forward -------------]
#                   |  eager  | compiled | compiled+cudagraphs |
# B=1,  S=128       |  3.2ms  |   2.1ms  |        1.1ms        |  ← CG wins at small batch
# B=4,  S=512       |  8.9ms  |   5.4ms  |        4.8ms        |
# B=16, S=512       | 28.3ms  |  15.2ms  |       14.9ms        |  ← fusion wins at large batch
# B=32, S=1024      | 84.1ms  |  41.7ms  |       41.2ms        |

The roofline model: is your kernel actually improvable?

Before optimizing a kernel, you must know whether it's worth optimizing. The roofline model answers this: given a kernel's arithmetic intensity, is it limited by the GPU's compute throughput (compute-bound) or its memory bandwidth (memory-bound)? This determines what kind of optimization can help.

Arithmetic intensity = FLOPs performed ÷ bytes of memory traffic. If this ratio is above the GPU's "ridge point" (compute peak ÷ memory bandwidth), the kernel is compute-bound. Below the ridge point, it's memory-bound.

For A100: 312 TFLOP/s ÷ 2 TB/s = 156 FLOP/byte ridge point. For T4: 65 TFLOP/s ÷ 320 GB/s = 203 FLOP/byte ridge point.

The roofline model's practical value is telling you which optimization matters:

For a memory-bound kernel (arithmetic intensity < ridge point): the path to speedup is reducing memory traffic. Fusion is the primary lever — instead of reading/writing each intermediate to VRAM, keep it in registers. Quantization (FP16→INT8) halves the bytes per element. Nothing you do to the arithmetic will help — the GPU is already waiting for data.

For a compute-bound kernel (arithmetic intensity > ridge point): the path to speedup is doing more arithmetic per clock cycle. Tensor Cores (BF16/FP16) deliver 16× more FLOP/s than CUDA cores. Better tiling that maximizes Tensor Core utilization. Nothing you do to reduce memory traffic will help — the GPU is already computing as fast as it can fetch data.

Calculating arithmetic intensity for real operations

To apply the roofline model, you need to compute the arithmetic intensity of the operation you're measuring. The formula is straightforward: count the FLOPs, count the bytes of memory traffic, divide.

def arithmetic_intensity(op_name, **shape):
    """Calculate FLOP count and byte count for common ops."""

    if op_name == "matmul":
        M, K, N = shape['M'], shape['K'], shape['N']
        flops = 2 * M * K * N         # each output: K multiplies + K-1 adds ≈ 2K FLOPs
        bytes_rw = (M*K + K*N + M*N) * 2  # read A, read B, write C in BF16
        return flops, bytes_rw, flops/bytes_rw

    elif op_name == "elementwise":
        N, flops_per_elem = shape['N'], shape.get('flops_per_elem', 1)
        flops   = N * flops_per_elem
        bytes_rw = N * 2 * 2          # read input + write output, BF16
        return flops, bytes_rw, flops/bytes_rw

    elif op_name == "attention":
        B, H, S, D = shape['B'], shape['H'], shape['S'], shape['D']
        # QK^T: [B,H,S,D] × [B,H,D,S] = [B,H,S,S], 2*B*H*S*S*D FLOPs
        flops_qk = 2 * B * H * S * S * D
        # Attn×V: [B,H,S,S] × [B,H,S,D] = [B,H,S,D], 2*B*H*S*S*D FLOPs
        flops_av = 2 * B * H * S * S * D
        flops    = flops_qk + flops_av
        # Bytes: read Q,K,V, write output, (plus the N² attn matrix in standard attn)
        bytes_qkv = B * H * S * D * 3 * 2       # Q, K, V tensors
        bytes_attn = B * H * S * S * 2          # attention scores (N² matrix)
        bytes_out  = B * H * S * D * 2
        bytes_rw  = bytes_qkv + bytes_attn + bytes_out
        return flops, bytes_rw, flops/bytes_rw

# A100 ridge point: 312e12 / 2e12 = 156 FLOP/byte
RIDGE_A100 = 156
RIDGE_T4   = 65e12 / 320e9   # ≈ 203 FLOP/byte

# Examples
for name, kwargs in [
    ("MatMul 4096x4096",   dict(op_name="matmul", M=4096, K=4096, N=4096)),
    ("MatMul 1x512x512",   dict(op_name="matmul", M=1, K=512, N=512)),
    ("GeLU 4M elements",   dict(op_name="elementwise", N=4_000_000, flops_per_elem=8)),
    ("ReLU 4M elements",   dict(op_name="elementwise", N=4_000_000, flops_per_elem=1)),
    ("Attn B=1 S=512",     dict(op_name="attention", B=1, H=12, S=512, D=64)),
    ("Attn B=32 S=512",    dict(op_name="attention", B=32, H=12, S=512, D=64)),
]:
    flops, byt, ai = arithmetic_intensity(**kwargs)
    bound = "compute" if ai > RIDGE_A100 else "memory"
    print(f"{name:20}: AI={ai:6.0f} FLOP/byte → {bound}-bound on A100")

# Output:
# MatMul 4096x4096    : AI=  1365 FLOP/byte → compute-bound on A100
# MatMul 1x512x512    : AI=   682 FLOP/byte → compute-bound on A100
# GeLU 4M elements    : AI=     4 FLOP/byte → memory-bound on A100
# ReLU 4M elements    : AI=     0 FLOP/byte → memory-bound on A100
# Attn B=1 S=512      : AI=     6 FLOP/byte → memory-bound on A100
# Attn B=32 S=512     : AI=   192 FLOP/byte → compute-bound on A100

Case study: attention at different batch sizes

The attention operation is the canonical example of a kernel that transitions from memory-bound to compute-bound as batch size increases. This is because the arithmetic intensity of attention grows with batch size while the memory traffic grows more slowly — understanding why leads directly to a prediction about when torch.compile and when FlashAttention help.

For standard (unfused) attention at batch=1, sequence=512, 12 heads, head_dim=64: arithmetic intensity is approximately 6 FLOP/byte — deeply memory-bound on any GPU. The dominant cost is reading and writing the N×N attention score matrix. Fusion (Flash Attention) dramatically helps here because it eliminates the N×N VRAM accesses entirely.

For batch=32, same config: arithmetic intensity jumps to ~192 FLOP/byte — compute-bound on A100. The QK^T matmul dominates. Flash Attention still helps (saves memory, which is good for longer sequences and larger batches), but the throughput limit is now the Tensor Core throughput, not bandwidth. Further fusion work won't improve it.

# ─── Full attention benchmark across batch sizes ─────────────────────────────
import torch, torch.nn.functional as F
from torch.utils import benchmark

B_dim, H, S, D = 12, 64, 512, 512  # heads, head_dim, seq_len, model_dim

results = []
for batch in [1, 2, 4, 8, 16, 32]:
    q = torch.randn(batch, H, S, D//H, device='cuda', dtype=torch.bfloat16)
    k = torch.randn(batch, H, S, D//H, device='cuda', dtype=torch.bfloat16)
    v = torch.randn(batch, H, S, D//H, device='cuda', dtype=torch.bfloat16)

    # Standard attention (naive: materialises N×N matrix)
    results.append(benchmark.Timer(
        stmt="F.scaled_dot_product_attention(q, k, v)",
        globals={"F": F, "q": q, "k": k, "v": v},
        label="SDPA",
        sub_label=f"batch={batch}",
        description="sdpa",
    ).timeit(100))

    # Compute efficiency: TFLOP/s achieved
    flops, byt, ai = arithmetic_intensity(
        op_name="attention", B=batch, H=H, S=S, D=D//H
    )
    t_ms = results[-1].mean * 1000
    achieved_tflops = flops / (results[-1].mean * 1e12)
    bound = "compute" if ai > 156 else "memory "
    print(f"B={batch:2d}: {t_ms:6.2f}ms  AI={ai:5.0f} FLOP/B  {bound}-bound  {achieved_tflops:.1f}T FLOP/s achieved")

# Expected output (A100):
# B= 1: 0.21ms  AI=    6 FLOP/B  memory -bound  0.6T FLOP/s achieved
# B= 2: 0.23ms  AI=   12 FLOP/B  memory -bound  1.2T FLOP/s achieved
# B= 4: 0.31ms  AI=   24 FLOP/B  memory -bound  1.9T FLOP/s achieved
# B= 8: 0.45ms  AI=   48 FLOP/B  memory -bound  2.6T FLOP/s achieved
# B=16: 0.72ms  AI=   96 FLOP/B  memory -bound  3.2T FLOP/s achieved
# B=32: 1.01ms  AI=  192 FLOP/B  compute-bound  8.8T FLOP/s achieved
#
# Key insight: achieved TFLOP/s jumps 3-4x at B=32 when we become compute-bound
# and Tensor Cores can be fully utilized. Below that threshold, we're
# bandwidth-limited regardless of how good the computation is.
